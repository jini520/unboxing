```ts
/** OCR 텍스트(읽기순서 보존)에서 PII 줄을 제거. 다층 방어(ADR-038). 상품·금액 줄은 보존. */

export function detectResidualPII(text: string): string[];
```

### 마스킹 규칙 (다층 — 순서대로)
1. **구간 제거(주 방어)**: PII 섹션 헤더 줄(`배송지`·`배송 주소`·`받는사람`·`받는 분`·`수령인`·`수취인`·`수신인`·`배송정보`) 부터 **다음 안전 섹션 헤더**(`결제정보`·`결제 정보`·`지불 방법`·`주문금액`·`주문상품`·`상품 상세`·`상품정보`) 직전까지 **통째 제거**. 라벨 없는 이름·주소도 위치로 제거됨.
2. **패턴 정규식(안전망)** — 줄 단위 제거:
   - 전화: `010[-\s]?\d{3,4}[-\s]?\d{4}`, `\+82[\s]?0?1\d{8,9}`, 마스킹형 `010[\*\s]*\d{3,4}`.
   - 우편번호: 5자리 숫자(괄호 포함 `(06234)` 형 포함).
   - 주소: 행정구역(`특별시|광역시|특별자치|도|시|군|구`) + 도로명(`...로|...길|번길`) 패턴 또는 `동 \d+호`.
   - 이메일·카드번호(`\d{4,6}[\*]{4,}\d{0,4}`).
3. **보수적 과삭제**: 분류엔 상품명·금액만 필요 → 애매한 PII 의심 줄은 제거(상품/금액 줄은 보존).
4. **잔존 게이트**: `detectResidualPII` 는 전화·이메일·우편번호·도로명 패턴 재매치를 반환. **마스킹 후 비어야 한다.**

## 금지사항

- 진짜 PII(실 이름·전화·주소·카드)를 fixture·테스트·repo 에 넣지 마라. 이유: 개인정보 비영속(CRITICAL) — 합성 데이터만.
- 구간 제거에만 의존하지 마라. 이유: 쿠팡형은 PII 섹션 헤더가 없어 구간이 안 먹는다(실측 — 패턴 안전망 필수, ADR-038).
- 상품명·금액 줄을 제거하지 마라. 이유: 분류 입력이 사라진다(과삭제는 PII 의심 줄에만).
- 구현을 먼저 쓰지 마라. 이유: test-first 필수(CLAUDE.md).
- 기존 테스트를 깨뜨리지 마라.
