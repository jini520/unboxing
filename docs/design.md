# 설계 상세 (Design Notes)

> CLAUDE.md 요약을 뒷받침하는 상세. 결정이 늘면 주제별 파일로 분리한다.

## 왜 이렇게 했는가 (다시 논의하기 전에 읽을 것)

- **순수 클라이언트가 아니라 서버리스 cron인 이유**: 모바일 OS는 앱이 종료/백그라운드일 때 "정확히 N분마다" 실행을 보장하지 않는다(iOS `BGAppRefreshTask`는 OS 재량, 폰이 꺼지면 미실행). "안 봐도 오는 알림"이 핵심 가치이므로 서버 측 폴링이 필수. "서버리스"는 "클라우드 없음"이 아니라 "관리할 상시 서버 없음".
- **공유 자격증명(내장)인 이유**: tracker.delivery Free는 단일 자격증명으로 $0이라 사용자별 키가 불필요. (스마트택배 시절의 per-key 쿼터 회피용 BYOK는 폐기.)
- **익명(기기 기반)인 이유**: 로그인 강제는 이탈만 늘림. 기기 토큰 + 송장만 저장. 기기 간 동기화는 Phase 2 계정으로.
- **D1인 이유**: cron이 Cloudflare Workers라 동일 생태계가 가장 단순. 계정/동기화가 일찍 필요하면 Supabase 대안.

## 운송장 등록 & 택배사 자동인식

- **택배사 식별 — 로컬 추정 + 확인**: tracker.delivery `track`은 `carrierId`가 필요하고 "번호→택배사 추천" 엔드포인트가 없음 → 운송장 번호 형식으로 후보를 로컬 추정해 제시, 사용자가 확인/수정, 실패 시 수동 드롭다운. (잘못된 택배사로 폴링하면 정보를 영영 못 잡으므로 확인 단계 유지)
- **미지원 택배사 폴백**: 등록 시 자동추정 택배사가 tracker.delivery 지원 목록에 없으면 → 자동 추적 대신 택배사 조회 페이지 **딥링크**만 제공 + "자동 추적 미지원" 안내. (스마트택배 재도입 X)
- **입력 진입 (MVP)**: 수동 입력 + 클립보드 감지(앱 열 때 운송장-형태 번호면 추가 제안). 바코드/QR·공유 익스텐션은 Phase 2.
- **등록 직후**: 즉시 1회 조회 → 정보 있으면 상태/타임라인, 없으면 "미등록" 안내 + 활성 폴링. (`등록` 알림은 첫 데이터 확인 시)
- **엣지**: 중복 번호 → 기존 항목 이동. 별명/메모는 Phase 2 검토.

## 상태 정규화 & 알림

택배사 상태 문구가 제각각이므로 표준 단계로 정규화 후, 단계 전환에만 알림.

| 표준 단계 | 의미 | 알림 |
|---|---|---|
| `등록` | 택배사에 송장 최초 확인 | ✅ |
| `집화` | 택배사 수거 | ✅ |
| `이동중` | 허브/간선 경유(여러 번) | ❌ 타임라인만 |
| `배송출발` | 오늘 도착 | ✅ |
| `배송완료` | 수령 | ✅ |
| `예외` | 지연·반송·주소불명 | ✅ |

- **과알림 방지**: `이동중`만 푸시 없이 타임라인. 그 외 전 단계 알림.
- **멱등성**: `last_normalized_status` 비교해 단계가 바뀔 때만 1회. 재독해도 중복 없음.
- **매핑 분리**: 원문 상태→표준 단계 매핑을 데이터(테이블)로. 택배사 추가 시 매핑만.
- **`등록` 알림 주의**: 폴링이 데이터를 처음 잡은 시점에 발송(앱 입력 순간 아님).

## 적응형 폴링 주기

**due 기반 단일 배치**: cron이 15분마다 떠서 `활성 AND now >= last_poll + interval(단계)` 인 송장만 폴링. cron 주기와 송장별 간격이 분리돼 API 호출은 due된 것만 나감(비용 중립).

단계별 간격:

| 단계 | 폴링 간격 |
|---|---|
| `미등록` | 6h |
| `등록`·`집화` | 4h |
| `이동중` | 4h |
| `배송출발` | 1h |
| `배송완료` | 중단 |
| `예외` | 12h |

- **만료/좀비**: 미등록 7일·예외 7일 자동 비활성. 등록 후 30일 지나도 완료/예외 아니면 강제 비활성(분실 의심)+알림.
- **배치 청크**: cron당 외부호출 50건(무료) 초과 시 다음 fire로 이월.
- **비용 검산**: 송장 1개 ~24회/평생(~10회/일). dedupe(동일 송장 공유)로 사용자 수 증가.

## 비용 모델 상세

병목은 컴퓨트가 아니라 택배 API 무료 쿼터.

| 층 | 무료 한도 | 한계 |
|---|---|---|
| Expo Push | 무제한 | 아님 |
| Cloudflare Workers/D1 | 10만 호출/일, cron당 외부호출 50건 | 여유 |
| tracker.delivery | Free 한도(미명시)→초과 시 Pro $49.99/Business $199.99·10만/월 | **진짜 천장** |

무료 한계 ≈ `API 일일 쿼터 ÷ (평균 활성 송장 × 송장당 하루 폴링)`. 스마트 폴링 시 수백 명.

스마트 폴링 핵심:
1. 배송완료 → 즉시 중단 (최대 절감)
2. 상태별 적응형 간격
3. 동일 송장 dedupe (여러 사용자 → API 1회 공유)
4. 배치 청크 (cron당 50건 초과분 이월)

## 데이터 모델 (Phase 1)

- `devices`: 푸시 토큰, 플랫폼, 등록 시각
- `shipments`: 택배사 코드, 운송장 번호, `last_normalized_status`, 마지막 폴링 시각, 활성 여부
- `subscriptions`: device ↔ shipment 다대다 (dedupe 폴링 근거)

## 개인정보 & 데이터 보존

**원칙: 개인정보는 영속 저장하지 않는다.** 유출돼도 털릴 개인정보가 없는 구조.

| 데이터 | D1 저장 |
|---|---|
| 운송장 번호·택배사 | ✅ (단독 식별 불가) |
| 정규화 상태·단계 시각·위치(허브명) | ✅ (개인정보 아님) |
| 푸시 토큰 | ✅ (전송 목적) |
| 수령인 이름·연락처·주소 | ❌ 조회 시 화면에만 표시 후 폐기 |

- **보관 기간 — 보관 안 함**: `배송완료` 감지 → 알림 발송 → 해당 송장 레코드 삭제. (완료 건은 앱에 남지 않음. 짧은 유예가 필요하면 추후 조정)
- **무효 토큰**: Expo가 invalid 알리면 즉시 삭제.
- **개인정보처리방침**(앱스토어 필수): 수집항목(운송장·푸시토큰)·목적·제3자 제공(tracker.delivery로 운송장 전송 + ToS상 추적데이터 일일 공유 의무)·국외이전(Cloudflare) 명시.

## 엣지 케이스 정책

| 상황 | 정책 |
|---|---|
| 미등록/오타 운송장 | 활성 폴링(6h), 7일 안 잡히면 자동 비활성 + "번호 확인" 알림 |
| API 일시 오류(5xx·타임아웃) | 조용히 다음 cron 재시도, `last_poll` 미갱신(백오프). 사용자 비노출 |
| 쿼터 소진(429/일 한도) | 폴링 일시 중단(리셋까지), `배송출발` 우선 처리. 지속 시 상위 플랜 검토 |
| 미매핑 원문 상태 | 안전 폴백 `기타`(무알림) + 로그(매핑 보강용). 오발송/크래시 금지 |
| 무효 푸시 토큰(DeviceNotRegistered) | 토큰 즉시 삭제 |
| 중복/경쟁 실행 | 상태 비교+갱신을 원자적으로 → 멱등(`last_normalized_status`) |
| 푸시 권한 거부 | 등록·인앱 조회는 가능, 알림 불가 명시 + 재요청 안내 |
| 좀비(30일 정체) | 강제 비활성 + 알림 |

## MVP 스코프 (Phase 1)

**포함**: 운송장 등록(수동+클립보드), 택배사 자동인식+확인(실패 시 수동), 송장 목록/상세 타임라인, 서버리스 cron 폴링 + 표준 단계 정규화, 푸시 알림(등록·집화·배송출발·배송완료·예외), 수동 삭제·당겨서 새로고침, 푸시 권한 온보딩, **다크모드**, 개인정보처리방침.

**제외 (Phase 2+)**: 해외(17TRACK), 계정/기기 간 동기화, 바코드/QR·공유 익스텐션, **별명/메모(Phase 2 결정)**, 홈 위젯, 통계/이력 분석, 다국어.
