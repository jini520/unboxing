# Architecture Decision Records

> 결정/이유/트레이드오프를 기록한다. 새 결정이 생기면 ADR을 추가하고, 뒤집힌 결정은 **Superseded** 표시 후 새 ADR로 대체한다.

## 철학

- **운영 서버 비용 0원** — 상시 서버 금지. 무료 티어 서버리스(클라우드 함수+스케줄러)만 사용.
- **마찰 최소** — 일반 대중 타깃. 회원가입·API 키 입력을 기본 경로에 넣지 않는다.
- **개인정보 비영속** — 유출돼도 털릴 개인정보가 없는 구조. 운송장·푸시토큰만 저장.
- **병목은 컴퓨트가 아니라 택배 API 쿼터** — 단, 아래 비용 모델에서 보듯 무료 한도가 넉넉해 실질 천장은 높다. 그래도 모든 설계는 외부 호출을 아끼는 방향을 유지한다.

### 비용 모델 (정정됨 — 2026-06 검증)

> ⚠️ 기존 문서는 무료 천장을 "수백 명"으로 봤으나, **과금 단위가 '호출 수'가 아니라 '월 고유 운송장 수'**임을 확인하여 정정한다.

| 층 | 무료 한도 | 한계 |
|---|---|---|
| Expo Push | 무제한 (배치 send 100/req, receipt 1000/req, 600/s) | 아님 |
| Cloudflare Workers/D1 | 10만 호출/일, **cron 1회 실행당 subrequest ≤50** | cron당 subrequest가 진짜 제약 |
| tracker.delivery | Free: **월 고유 운송장 ~100k/프로젝트** (신용카드 불필요), 고유 운송장·활성 webhook 수 모두 **요청 시 무료 확장** | 자격증명 21일 만료(아래 ADR-013) |

- **과금 단위 = 월 고유 운송장 수.** 같은 송장을 하루 10번 폴링해도 그 달의 고유 카운트는 1. 따라서 폴링 *빈도*는 월 쿼터에 영향이 거의 없다.
- **실질 무료 천장 ≈ 수천~만 명** (평균 1인 10송장/월 가정 시 월 고유 ~100k → 약 1만 명). "수백 명"이 아니다.
- **여전히 아껴야 하는 것**: ① cron 1회 subrequest ≤50 (그래서 청크/이월 필요) ② API 예의(같은 송장 과도 폴링 자제) ③ 신선도(불필요한 헛폴링 제거).
- **진짜 절감 레버는 dedupe** — 여러 사용자가 같은 송장을 봐도 *고유 운송장 1개*로 카운트.

> 미검증 항목은 문서 끝 **Open Questions** 참조 (콘솔에서 확정 검증 필요).

### ADR-001: 순수 클라이언트가 아니라 서버리스 cron 폴링
**결정**: 배송 추적 폴링을 Cloudflare Workers Cron(서버 측)에서 수행.
**이유**: 모바일 OS는 앱이 종료/백그라운드일 때 "정확히 N분마다" 실행을 보장하지 않는다(iOS `BGAppRefreshTask`는 OS 재량, 폰이 꺼지면 미실행). "안 봐도 오는 알림"이 핵심 가치이므로 서버 측 폴링 필수. "서버리스" = "클라우드 없음"이 아니라 "관리할 상시 서버 없음".
**트레이드오프**: 무료 서버리스 함수·스케줄러에 종속. 클라이언트-온리 대비 백엔드 운영 필요(단, 무료).

### ADR-002: 익명(기기 기반), 로그인 없음
**결정**: 계정 없이 기기 푸시 토큰 + 송장만 저장.
**이유**: 로그인 강제는 이탈만 늘림(마찰 최소 철학).
**트레이드오프**: 기기 간 동기화 불가 → Phase 2 계정으로 미룸. 인증을 secret device_id로 대체(ADR-007).

### ADR-003: Cloudflare D1 + Workers
**결정**: 저장소로 Cloudflare D1(SQLite)을 사용.
**이유**: cron이 Cloudflare Workers라 동일 생태계가 가장 단순(무료).
**트레이드오프**: 계정/동기화가 일찍 필요해지면 Supabase 등 재검토 여지.

### ADR-004: 공유 자격증명 내장 (BYOK 폐기)
**결정**: tracker.delivery 자격증명을 서버에 시크릿으로 내장(사용자별 키 없음).
**이유**: 단일 자격증명으로 충분하고 사용자별 키는 마찰만 늘림. (스마트택배 시절의 per-key 쿼터 회피용 BYOK 폐기.)
**트레이드오프**: 전체 사용자가 하나의 쿼터를 공유 → dedupe·고유-번호 과금 구조로 방어. 자격증명 노출 시 영향 큼 → 시크릿 관리 엄격(ARCHITECTURE 보안).

### ADR-005: 개인정보 비영속
**결정**: 수령인 이름·연락처·주소를 D1에 저장하지 않고 조회 시 화면에만 표시 후 폐기.
**이유**: 유출 리스크 자체를 제거. 운송장·택배사·정규화 상태는 단독으로 개인 식별 불가. 허브명(위치)은 비개인정보로 간주.
**트레이드오프**: 개인정보처리방침에 수집항목·제3자 제공(tracker.delivery로 운송장 전송)·국외이전(Cloudflare) 명시 필요.

> **개정(2026-06-17)**: `배송완료` 시 **즉시 자동 삭제**하던 정책을 폐기한다. **기본 사양 = 보관** — 완료 시 `last_normalized_status='배송완료'` + `active=0`(재폴링 중단)으로 두고, **사용자가 직접 삭제**(목록 스와이프 / `DELETE /shipments/:id`)한다. 알림은 종전대로 전환 시 1회. 보관하는 데이터는 운송장·택배사·정규화 상태뿐(수령인 PII 아님)이라 본 ADR의 핵심(수령인 정보 비영속)과 무관하다. **자동 삭제는 옵트인 설정으로 다음 phase 제공** → `docs/ROADMAP.md`. (완료 건 이력이 앱에 남아 사용자가 확인·정리 가능.)

### ADR-006: 적응형 폴링 (정정 — 근거 재서술)
**결정**: 원문 상태를 표준 7단계로 정규화하고, 단계별 적응형 간격으로 due된 송장만 폴링.
**이유 (정정됨)**: ~~월 API 쿼터 절감~~ 이 아니다(과금=월 고유 운송장 수라 빈도는 월 쿼터와 무관). 적응형 폴링의 실제 근거는 **① cron 1회 subrequest ≤50 한도 분산, ② 신선도(배송출발은 1h, 정체 단계는 길게), ③ API 예의(불필요한 헛폴링 억제)**. 완료 시 폴링 중단도 같은 이유.
**트레이드오프**: 정규화 매핑 유지보수 필요(ADR-009). 간격이 길수록 알림 지연 ↔ 호출 절약. *dedupe*는 별개로 월 고유-번호 절감에 직접 기여하므로 유지.

### ADR-007: HTTP API + 디바이스 식별/인가
**결정**: 클라이언트가 최초 실행 시 고엔트로피 secret `device_id`(랜덤 ≥128bit, 예: `crypto.randomUUID()`×2 또는 32바이트 base64url)를 생성·기기 보관하고, 모든 API 호출에 `Authorization: Bearer <device_id>`로 전송. 인가는 `subscriptions(device_id, shipment_id)` 소유권으로 판단(자기 구독만 조회/삭제).
**이유**: 계정 없는(ADR-002) 상태에서 "내 송장만" 보장하려면 추측 불가한 비밀이 곧 자격이어야 한다. 별도 로그인 0.
**트레이드오프**: `device_id` 유출=해당 기기 구독 전체 노출(단, 개인정보 비영속이라 운송장/상태뿐). 앱 삭제/초기화 시 device_id 소실 → 구독 복구 불가(Phase 2 계정 동기화로 해결). device_id는 절대 로그에 남기지 않는다.

### ADR-008: 공개 API 남용 방어 (쿼터·subrequest 보호)
**결정**: 인증이 약한 공개 API를 **서버측 silent throttle**로 방어 — 디바이스/IP별 등록 레이트 제한 + **디바이스당 활성 송장 상한**(예: 100). 초과 시 429. CAPTCHA(Cloudflare Turnstile)는 도입하지 않는다.
**이유**: 익명·무마찰이라 누구나 호출 가능 → 악의적 대량 등록이 고유-운송장 쿼터/subrequest를 소진할 수 있다. silent throttle은 정상 사용자에게 **마찰 0**(철학 유지). CAPTCHA는 마찰을 주므로 회피.
**트레이드오프**: 분산 대량 공격엔 완전 방어 못 함 → 지속 시 Turnstile/Cloudflare WAF rate limit를 **Phase 2 에스컬레이션**으로 둔다. IP 기반은 NAT/모바일 공유 IP에서 거짓양성 가능 → 디바이스 기준을 우선.

### ADR-009: 상태 정규화 매핑은 코드(Phase 1)
**결정**: 원문 상태 코드 → 표준 7단계 매핑을 Phase 1에서는 **코드(상수 맵)**로 둔다. D1 매핑 테이블은 도입하지 않는다.
**이유**: Phase 1은 국내 소수 택배사 + tracker.delivery 표준 status 코드(`DELIVERED` 등) 중심이라 매핑이 작고 안정적. 코드 맵이 가장 단순하고 테스트(전수)가 쉽다.
**트레이드오프**: 택배사/원문 상태가 급증하면 배포 없이 못 바꿈 → 그 시점에 **D1 매핑 테이블로 이전**(Phase 2). 미매핑 원문은 안전 폴백 `기타`(무알림)+로그로 흡수.

### ADR-010: Expo Push 2단계(ticket → receipt) 처리
**결정**: 푸시를 **send(→ticket) → ~15분 후 getReceipts(→receipt)** 2단계로 처리. send는 배치 100건, receipt 조회는 배치 1000건. `DeviceNotRegistered`(ticket 또는 receipt)면 해당 토큰 즉시 삭제. `MessageRateExceeded`/5xx는 지수 백오프 재시도.
**이유**: ticket은 "Expo가 수신"일 뿐 실제 FCM/APNs 전달 성공이 아니다. 무효 토큰은 주로 receipt에서 드러나므로 receipt를 확인해야 토큰 위생을 유지하고 헛발송을 막는다.
**트레이드오프**: receipt 확인용 별도 cron 작업 + ticket id 임시 보관 필요. 약간의 상태(보낸 ticket) 저장 → 발송 직후 삭제로 최소화.
**보안**: 서버 측 발송 인증은 Expo Enhanced Security(access token) 사용을 권장 — 활성화 시 `EXPO_ACCESS_TOKEN` secret 필수(→ ARCHITECTURE 환경변수 섹션).

### ADR-011: 타임라인/이벤트는 저장하지 않고 실시간 조회
**결정**: 상세 화면 타임라인은 D1에 저장하지 않고, 상세 진입 시 `track`을 실시간 조회해 표시. D1에는 `last_normalized_status`/`last_polled_at` 등 폴링·알림에 필요한 최소 상태만 둔다.
**이유**: 개인정보 비영속(ADR-005)과 저장 절약. 타임라인 원문에는 위치/시각 등 식별성 낮은 정보가 섞이나, 저장하지 않으면 유출면이 없다.
**트레이드오프**: 상세 진입 시 1회 API 호출(고유-번호 과금엔 영향 없음, subrequest는 사용자 행동이라 분산됨). 오프라인 시 상세 타임라인은 마지막 캐시만 표시(ADR-014).

### ADR-012: cron 단일 실행/중첩 방지 + KST 시간대
**결정**: cron(15분)은 짧게 끝내도록 청크(subrequest ≤50)로 제한하고, 한 실행이 다음 fire를 넘기지 않게 한다. 중첩 우려 작업은 처리 시작 시 `last_polled_at`를 선점 갱신해 같은 송장 중복 처리를 막는다. "오늘 도착(배송출발)" 등 날짜 판정은 **KST(UTC+9)** 기준.
**이유**: Workers cron은 시간당 정시 트리거지만 실행이 길어지면 겹칠 수 있다. 날짜 경계가 어긋나면 "오늘 도착" 알림이 오발송된다.
**트레이드오프**: 선점 갱신은 폴링 실패 시 백오프 로직과 조심히 결합해야 한다(실패면 `last_polled_at` 롤백/미갱신). 분산 락 없이 D1 원자 갱신에 의존.

### ADR-013: tracker.delivery 토큰 캐싱 + Free 자격증명 21일 수동 재발급 수용
**결정**: client_credentials로 받은 **access token은 D1에 캐싱**하고 만료 전 cron이 자동 재발급. Free 플랜의 **client 자격증명(21일 만료)은 콘솔 수동 재발급**을 수용하고, 만료 임박 알림(로그/운영자 통지)을 둔다. 재발급이 운영 부담이 되면 **Pro(무만료, 유료)**로 전환.
**이유**: 토큰 자동 갱신은 무인 운영을 유지. 그러나 Free 자격증명 재발급이 API로 자동화 가능한지는 미확인(Open Questions) → 최악의 경우 21일마다 수동 1회를 받아들이는 게 현실적. 이것이 "$0 무인 운영"의 유일한 균열.
**트레이드오프**: 21일마다 수동 작업 누락 시 폴링 전면 중단 위험 → 만료 7일 전부터 경고. Pro 전환은 $0 제약을 깨므로 최후 수단.

### ADR-014: 앱은 서버를 SOT로, 로컬은 캐시만
**결정**: 송장 목록/상태의 진실 원천(SOT)은 서버(D1). 앱은 마지막 응답을 로컬 캐시로 두어 **오프라인 읽기**만 지원하고, 변경(등록/삭제)은 온라인에서만.
**이유**: 동기화 충돌을 피하고 단순. 오프라인에서도 "마지막 상태"는 봐야 한다.
**트레이드오프**: 오프라인 중 등록/삭제 큐잉은 하지 않음(Phase 2). 캐시와 서버 불일치 가능 → 당겨서 새로고침으로 해소.

### ADR-015: 폴링-first + webhook은 검증된 최적화/폴백 구조
**결정**: Phase 1은 **폴링을 1차 메커니즘**으로 한다. tracker.delivery **Webhook(`registerTrackWebhook`)**은 "검증된 권장 최적화"로 ARCHITECTURE에 설계해두되(콜백 수신 + 24h 재등록 sweep), 도입 시에도 **폴링을 폴백으로 유지**.
**이유**: 과금이 월 고유-운송장 수라 **webhook은 월 쿼터를 절감하지 않는다**(같은 고유 송장). 이득은 신선도↑·cron subrequest↓·헛폴링 제거뿐인데, 24h 재등록·콜백 보안 복잡도가 따른다. 비용이 천장이 아닌 현 상황에선 폴링-first가 더 단순·견고.
**트레이드오프**: 실시간성이 중요해지거나 subrequest 압박이 커지면 webhook 우선 전환. Free에서 webhook 사용 가능 여부 미확인(Open Questions) → 검증 후 활성화.

### ADR-016: 테마 — 시스템 외형 따름, 라이트 기준 설계 (다크 지원)
**결정**: 앱은 **시스템 외형(light/dark)을 따른다**. 디자인 기준선(baseline)은 **라이트**이며 다크는 동등 품질로 지원한다. "다크모드 고정"은 폐기. (`app.json`도 이미 `userInterfaceStyle`를 자동/시스템으로 둔다 — 현재 `light` 고정값을 시스템 추종으로 조정.)
**이유**: 대중 소비자용 택배 추적 앱은 라이트가 표준 기대치(국내 택배사/배송 앱 관행). 다크 전용/다크 디폴트는 이질적이고 신뢰감을 떨어뜨린다. 단 다크 선호 사용자를 위해 다크는 지원해야 한다(원래 MVP 요구의 "다크모드"는 *전용*이 아니라 *지원*으로 재해석).
**트레이드오프**: 라이트/다크 두 팔레트를 다 관리해야 함 → 색을 하드코딩하지 않고 **시맨틱 토큰**으로 추상화(UI_GUIDE). 초기 디자인 노력 ↑.

### ADR-017: 계정 없는 "데이터 삭제" 경로 제공 (스토어 정책 충족)
**결정**: 계정은 없지만(ADR-002) **인앱 "모든 데이터 삭제"**를 설정 화면에 명확히 제공 — `device_id` + 구독 + (orphan) 송장 + 푸시 토큰을 서버에서 폐기(`DELETE /me`). 보조로 개인정보처리방침에 데이터 삭제/문의 경로 명시.
**이유**: Apple 5.1.1(v)와 Google Play User Data 정책은 데이터 수집 앱에 **삭제 경로**를 요구한다(계정 유무와 별개로 데이터 삭제/동의 철회 수단 필요, 설정에 눈에 띄게). 미제공 시 심사 거절·삭제 리스크.
**트레이드오프**: 삭제 후 복구 불가(개인정보 비영속과 일관). 삭제는 즉시·완전(법적 보존 의무 없음). UX상 확인 다이얼로그 필수.

### ADR-018: 알림은 거래성(transactional)만 — 광고성 정보 규제 회피
**결정**: 푸시는 **배송 상태 변화(거래성/정보성) 알림으로 한정**. 마케팅·프로모션 푸시는 Phase 1에서 발송하지 않는다.
**이유**: 정보통신망법상 **영리 목적 광고성 정보**는 사전 수신동의 + `[광고]` 표기 + 야간(21시~익일 08시) 별도 동의가 필요하다. 거래성 알림은 이에 해당하지 않아 규제를 피하고 마찰을 없앤다. Apple/Google도 푸시를 광고 용도로 강요하는 것을 금지한다.
**트레이드오프**: 마케팅 채널로 못 씀(Phase 2에 별도 동의·표기·야간 제한 도입 시 가능). 조용시간(PRD)은 과알림 방지 차원이며 법적 야간 제한과는 별개로 운영.

### ADR-019: App Review/심사용 데모 경로 제공
**결정**: 심사자가 **실제 배송 없이** 앱 기능을 검증할 수 있도록 — ① 캔드(canned) 응답을 돌려주는 **데모용 운송장 번호**(또는 데모 모드)와 ② 제출 시 **리뷰 노트**(샘플 번호·예상 동작)를 함께 제공.
**이유**: 추적 앱은 등록 직후 "미등록"만 보일 수 있어 심사자가 핵심 가치(상태 변화→알림)를 못 볼 위험 → minimum functionality(Apple 4.2)·기능 미확인 거절 방지.
**트레이드오프**: 데모 경로를 코드/데이터에 두어야 함(소량). 데모 번호는 실제 폴링을 타지 않도록 분기.

## Open Questions (콘솔/실측 검증 대기)

> 아래는 외부 문서(SPA)가 fetch되지 않아 **미검증**. tracker.delivery 콘솔/실측으로 확정한 뒤 관련 ADR을 업데이트한다.

1. **무료 월 고유 운송장 정확 수치** — "~100k"는 검색 스니펫 기준. 콘솔에서 정확값 확인. (영향: 비용 모델, ADR-006)
2. **"10,000" 수치의 의미** — Business 미만 플랜의 어떤 내장 쿼터인지 불명.
3. **초당/분당 rate limit 존재 여부** — 월 쿼터와 별개의 burst 제한이 있으면 cron 청크 크기 재조정 필요.
4. **Free 플랜에서 Webhook 사용 가능 여부** — 활성 webhook 수가 0이면 ADR-015 webhook 경로는 Phase 2로.
5. **Free 자격증명(21일) API 자동 재발급 가능 여부** — 가능하면 ADR-013의 "수동" 전제와 균열이 사라짐.
