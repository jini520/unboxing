# Architecture Decision Records

> 결정/이유/트레이드오프를 기록한다. 새 결정이 생기면 ADR을 추가하고, 뒤집힌 결정은 **Superseded** 표시 후 새 ADR로 대체한다.

## 철학

- **운영 서버 비용 0원** — 상시 서버 금지. 무료 티어 서버리스(클라우드 함수+스케줄러)만 사용.
- **마찰 최소** — 일반 대중 타깃. 회원가입·API 키 입력을 기본 경로에 넣지 않는다.
- **개인정보 비영속** — 유출돼도 털릴 개인정보가 없는 구조. 운송장·푸시토큰만 저장.
- **병목은 컴퓨트가 아니라 택배 API 쿼터** — 단, 아래 비용 모델에서 보듯 무료 한도가 넉넉해 실질 천장은 높다. 그래도 모든 설계는 외부 호출을 아끼는 방향을 유지한다.

### 비용 모델 (정정됨 — 2026-06 검증)

> ⚠️ 기존 문서는 무료 천장을 "수백 명"으로 봤으나, **과금 단위가 '호출 수'가 아니라 '월 고유 운송장 수'**임을 확인하여 정정한다.

| 층 | 무료 한도 | 한계 |
|---|---|---|
| Expo Push | 무제한 (배치 send 100/req, receipt 1000/req, 600/s) | 아님 |
| Cloudflare Workers/D1 | 10만 호출/일, **cron 1회 실행당 subrequest ≤50** | cron당 subrequest가 진짜 제약 |
| tracker.delivery | Free(검증 2026-06): **월 고유 운송장 1만/프로젝트**, **활성 webhook 1000 동시**, **10 req/s** burst. 고유 운송장·활성 webhook 수 모두 **요청 시 무료 확장**(신용카드 불필요) | 자격증명 21일 만료(아래 ADR-013) |

- **과금 단위 = 월 고유 운송장 수.** 같은 송장을 하루 10번 폴링해도 그 달의 고유 카운트는 1. 따라서 폴링 *빈도*는 월 쿼터에 영향이 거의 없다.
- **실질 무료 천장 ≈ 수백~천 명** (검증된 월 고유 1만 기준 — 평균 1인 10송장/월이면 ~1천 명, 1인 1송장이면 ~1만 명). 과거 "~100k" 추정은 오류였고, 필요 시 **무료 확장 요청**으로 상향한다("수백 명" 비관도 아니다).
- **여전히 아껴야 하는 것**: ① cron 1회 subrequest ≤50 (그래서 청크/이월 필요) ② API 예의(같은 송장 과도 폴링 자제) ③ 신선도(불필요한 헛폴링 제거) ④ **10 req/s burst** — webhook 콜백·cron 청크가 초당 한도를 넘지 않게(콜백은 비동기 track, → ADR-028).
- **진짜 절감 레버는 dedupe** — 여러 사용자가 같은 송장을 봐도 *고유 운송장 1개*로 카운트(활성 webhook도 송장당 1개라 1000 동시 한도 절감에 동일하게 기여, → ADR-028).

> 미검증 항목은 문서 끝 **Open Questions** 참조 (콘솔에서 확정 검증 필요).

### ADR-001: 순수 클라이언트가 아니라 서버리스 cron 폴링
**결정**: 배송 추적 폴링을 Cloudflare Workers Cron(서버 측)에서 수행.
**이유**: 모바일 OS는 앱이 종료/백그라운드일 때 "정확히 N분마다" 실행을 보장하지 않는다(iOS `BGAppRefreshTask`는 OS 재량, 폰이 꺼지면 미실행). "안 봐도 오는 알림"이 핵심 가치이므로 서버 측 폴링 필수. "서버리스" = "클라우드 없음"이 아니라 "관리할 상시 서버 없음".
**트레이드오프**: 무료 서버리스 함수·스케줄러에 종속. 클라이언트-온리 대비 백엔드 운영 필요(단, 무료).

### ADR-002: 익명(기기 기반), 로그인 없음
**결정**: 계정 없이 기기 푸시 토큰 + 송장만 저장.
**이유**: 로그인 강제는 이탈만 늘림(마찰 최소 철학).
**트레이드오프**: 기기 간 동기화 불가 → Phase 2 계정으로 미룸. 인증을 secret device_id로 대체(ADR-007).

### ADR-003: Cloudflare D1 + Workers
**결정**: 저장소로 Cloudflare D1(SQLite)을 사용.
**이유**: cron이 Cloudflare Workers라 동일 생태계가 가장 단순(무료).
**트레이드오프**: 계정/동기화가 일찍 필요해지면 Supabase 등 재검토 여지.

### ADR-004: 공유 자격증명 내장 (BYOK 폐기)
**결정**: tracker.delivery 자격증명을 서버에 시크릿으로 내장(사용자별 키 없음).
**이유**: 단일 자격증명으로 충분하고 사용자별 키는 마찰만 늘림. (스마트택배 시절의 per-key 쿼터 회피용 BYOK 폐기.)
**트레이드오프**: 전체 사용자가 하나의 쿼터를 공유 → dedupe·고유-번호 과금 구조로 방어. 자격증명 노출 시 영향 큼 → 시크릿 관리 엄격(ARCHITECTURE 보안).

### ADR-005: 개인정보 비영속
**결정**: 수령인 이름·연락처·주소를 D1에 저장하지 않고 조회 시 화면에만 표시 후 폐기.
**이유**: 유출 리스크 자체를 제거. 운송장·택배사·정규화 상태는 단독으로 개인 식별 불가. 허브명(위치)은 비개인정보로 간주.
**트레이드오프**: 개인정보처리방침에 수집항목·제3자 제공(tracker.delivery로 운송장 전송)·국외이전(Cloudflare) 명시 필요.

> **개정(2026-06-17)**: `배송완료` 시 **즉시 자동 삭제**하던 정책을 폐기한다. **기본 사양 = 보관** — 완료 시 `last_normalized_status='배송완료'` + `active=0`(재폴링 중단)으로 두고, **사용자가 직접 삭제**(목록 스와이프 / `DELETE /shipments/:id`)한다. 알림은 종전대로 전환 시 1회. 보관하는 데이터는 운송장·택배사·정규화 상태뿐(수령인 PII 아님)이라 본 ADR의 핵심(수령인 정보 비영속)과 무관하다. **자동 삭제는 옵트인 설정으로 다음 phase 제공** → `docs/ROADMAP.md`. (완료 건 이력이 앱에 남아 사용자가 확인·정리 가능.)

> **수취인 패스스루(2026-06-17)**: 상세 화면용 수취인 정보(이름·지역명)는 `GET /shipments/:id` 의 실시간 track 응답을 **화면 전용으로 패스스루만** 한다(미저장) — 본 ADR 핵심(비영속)과 일관. `phoneNumber` 는 받지 않고, recipient/sender 를 D1 어떤 테이블에도 INSERT/UPDATE 하지 않으며 로그에도 남기지 않는다. `tryTrack` 이 null(자격증명 없음·외부 실패)이면 `recipient: null`.

### ADR-006: 적응형 폴링 (정정 — 근거 재서술)
**결정**: 원문 상태를 표준 7단계로 정규화하고, 단계별 적응형 간격으로 due된 송장만 폴링.
**이유 (정정됨)**: ~~월 API 쿼터 절감~~ 이 아니다(과금=월 고유 운송장 수라 빈도는 월 쿼터와 무관). 적응형 폴링의 실제 근거는 **① cron 1회 subrequest ≤50 한도 분산, ② 신선도(배송출발은 1h, 정체 단계는 길게), ③ API 예의(불필요한 헛폴링 억제)**. 완료 시 폴링 중단도 같은 이유.
**트레이드오프**: 정규화 매핑 유지보수 필요(ADR-009). 간격이 길수록 알림 지연 ↔ 호출 절약. *dedupe*는 별개로 월 고유-번호 절감에 직접 기여하므로 유지.
> **개정(2026-06-26 · ADR-028)**: webhook-first 전환으로 적응형 폴링은 **1차 신선도 메커니즘에서 저빈도 폴백으로 강등**된다(놓친 콜백·등록 실패·1000 동시 초과분 대비). 단계별 간격(배송출발 1h 등)은 폴백 cadence(~12h 단일)로 완화 — 신선도는 webhook 담당. 정규화·CAS·dedupe 로직은 그대로 재사용(콜백도 같은 다운스트림).

### ADR-007: HTTP API + 디바이스 식별/인가
**결정**: 클라이언트가 최초 실행 시 고엔트로피 secret `device_id`(랜덤 ≥128bit, 예: `crypto.randomUUID()`×2 또는 32바이트 base64url)를 생성·기기 보관하고, 모든 API 호출에 `Authorization: Bearer <device_id>`로 전송. 인가는 `subscriptions(device_id, shipment_id)` 소유권으로 판단(자기 구독만 조회/삭제).
**이유**: 계정 없는(ADR-002) 상태에서 "내 송장만" 보장하려면 추측 불가한 비밀이 곧 자격이어야 한다. 별도 로그인 0.
**트레이드오프**: `device_id` 유출=해당 기기 구독 전체 노출(단, 개인정보 비영속이라 운송장/상태뿐). 앱 삭제/초기화 시 device_id 소실 → 구독 복구 불가(Phase 2 계정 동기화로 해결). device_id는 절대 로그에 남기지 않는다.

### ADR-008: 공개 API 남용 방어 (쿼터·subrequest 보호)
**결정**: 인증이 약한 공개 API를 **서버측 silent throttle**로 방어 — 디바이스/IP별 등록 레이트 제한 + **디바이스당 활성 송장 상한**(예: 100). 초과 시 429. CAPTCHA(Cloudflare Turnstile)는 도입하지 않는다.
**이유**: 익명·무마찰이라 누구나 호출 가능 → 악의적 대량 등록이 고유-운송장 쿼터/subrequest를 소진할 수 있다. silent throttle은 정상 사용자에게 **마찰 0**(철학 유지). CAPTCHA는 마찰을 주므로 회피.
**트레이드오프**: 분산 대량 공격엔 완전 방어 못 함 → 지속 시 Turnstile/Cloudflare WAF rate limit를 **Phase 2 에스컬레이션**으로 둔다. IP 기반은 NAT/모바일 공유 IP에서 거짓양성 가능 → 디바이스 기준을 우선.

### ADR-009: 상태 정규화 매핑은 코드(Phase 1)
**결정**: 원문 상태 코드 → 표준 7단계 매핑을 Phase 1에서는 **코드(상수 맵)**로 둔다. D1 매핑 테이블은 도입하지 않는다.
**이유**: Phase 1은 국내 소수 택배사 + tracker.delivery 표준 status 코드(`DELIVERED` 등) 중심이라 매핑이 작고 안정적. 코드 맵이 가장 단순하고 테스트(전수)가 쉽다.
**트레이드오프**: 택배사/원문 상태가 급증하면 배포 없이 못 바꿈 → 그 시점에 **D1 매핑 테이블로 이전**(Phase 2). 미매핑 원문은 안전 폴백 `기타`(무알림)+로그로 흡수.

### ADR-010: Expo Push 2단계(ticket → receipt) 처리
**결정**: 푸시를 **send(→ticket) → ~15분 후 getReceipts(→receipt)** 2단계로 처리. send는 배치 100건, receipt 조회는 배치 1000건. `DeviceNotRegistered`(ticket 또는 receipt)면 해당 토큰 즉시 삭제. `MessageRateExceeded`/5xx는 지수 백오프 재시도.
**이유**: ticket은 "Expo가 수신"일 뿐 실제 FCM/APNs 전달 성공이 아니다. 무효 토큰은 주로 receipt에서 드러나므로 receipt를 확인해야 토큰 위생을 유지하고 헛발송을 막는다.
**트레이드오프**: receipt 확인용 별도 cron 작업 + ticket id 임시 보관 필요. 약간의 상태(보낸 ticket) 저장 → 발송 직후 삭제로 최소화.
**보안**: 서버 측 발송 인증은 Expo Enhanced Security(access token) 사용을 권장 — 활성화 시 `EXPO_ACCESS_TOKEN` secret 필수(→ ARCHITECTURE 환경변수 섹션).

### ADR-011: 타임라인/이벤트는 저장하지 않고 실시간 조회
**결정**: 상세 화면 타임라인은 D1에 저장하지 않고, 상세 진입 시 `track`을 실시간 조회해 표시. D1에는 `last_normalized_status`/`last_polled_at` 등 폴링·알림에 필요한 최소 상태만 둔다.
**이유**: 개인정보 비영속(ADR-005)과 저장 절약. 타임라인 원문에는 위치/시각 등 식별성 낮은 정보가 섞이나, 저장하지 않으면 유출면이 없다.
**트레이드오프**: 상세 진입 시 1회 API 호출(고유-번호 과금엔 영향 없음, subrequest는 사용자 행동이라 분산됨). 오프라인 시 상세 타임라인은 마지막 캐시만 표시(ADR-014).

### ADR-012: cron 단일 실행/중첩 방지 + KST 시간대
**결정**: cron(15분)은 짧게 끝내도록 청크(subrequest ≤50)로 제한하고, 한 실행이 다음 fire를 넘기지 않게 한다. 중첩 우려 작업은 처리 시작 시 `last_polled_at`를 선점 갱신해 같은 송장 중복 처리를 막는다. "오늘 도착(배송출발)" 등 날짜 판정은 **KST(UTC+9)** 기준.
**이유**: Workers cron은 시간당 정시 트리거지만 실행이 길어지면 겹칠 수 있다. 날짜 경계가 어긋나면 "오늘 도착" 알림이 오발송된다.
**트레이드오프**: 선점 갱신은 폴링 실패 시 백오프 로직과 조심히 결합해야 한다(실패면 `last_polled_at` 롤백/미갱신). 분산 락 없이 D1 원자 갱신에 의존.
> **개정(2026-06-26 · ADR-028)**: cron(15~30분)은 **유지보수 + 저빈도 폴백**으로 역할 전환 — ① receipt sweep ② 아침 flush ③ **webhook 24h 재등록 sweep** ④ **lifecycle 독립 sweep**(미등록7일·분실30일 — 폴링 루프에서 **분리**: webhook은 "변화 없음"을 감지 못해 폴링 안에서 판정하면 누락) ⑤ 저빈도 폴백 재폴링 ⑥ 보존·rate_limits 정리. cron *fire* 빈도는 무료(진짜 비용은 fire당 subrequest≤50·10 req/s)라 유지하되, 송장별 폴백 재폴링 간격만 늘려 subrequest를 줄인다.

### ADR-013: tracker.delivery 토큰 캐싱 + Free 자격증명 21일 수동 재발급 수용
**결정**: client_credentials로 받은 **access token은 D1에 캐싱**하고 만료 전 cron이 자동 재발급. Free 플랜의 **client 자격증명(21일 만료)은 콘솔 수동 재발급**을 수용하고, 만료 임박 알림(로그/운영자 통지)을 둔다. 재발급이 운영 부담이 되면 **Pro(무만료, 유료)**로 전환.
**이유**: 토큰 자동 갱신은 무인 운영을 유지. 그러나 Free 자격증명 재발급이 API로 자동화 가능한지는 미확인(Open Questions) → 최악의 경우 21일마다 수동 1회를 받아들이는 게 현실적. 이것이 "$0 무인 운영"의 유일한 균열.
**트레이드오프**: 21일마다 수동 작업 누락 시 폴링 전면 중단 위험 → 만료 7일 전부터 경고. Pro 전환은 $0 제약을 깨므로 최후 수단.

### ADR-014: 앱은 서버를 SOT로, 로컬은 캐시만
**결정**: 송장 목록/상태의 진실 원천(SOT)은 서버(D1). 앱은 마지막 응답을 로컬 캐시로 두어 **오프라인 읽기**만 지원하고, 변경(등록/삭제)은 온라인에서만.
**이유**: 동기화 충돌을 피하고 단순. 오프라인에서도 "마지막 상태"는 봐야 한다.
**트레이드오프**: 오프라인 중 등록/삭제 큐잉은 하지 않음(Phase 2). 캐시와 서버 불일치 가능 → 당겨서 새로고침으로 해소.

### ADR-015: 폴링-first + webhook은 검증된 최적화/폴백 구조 — **Superseded by ADR-028 (2026-06-26)**
> **뒤집힘(2026-06-26)**: Open Q4(Free webhook 지원)가 검증됨 — Free에서 **활성 webhook 1000 동시·무료 확장** 확인. ADR-015의 "검증 후 활성화" 조건이 충족되었고, 트레이드오프에 적힌 "실시간성이 중요해지면 webhook 우선 전환"을 실행한다. **신결정 → ADR-028(webhook-first), 콜백 인증 → ADR-029.** 아래 원결정은 이력으로 보존.

**결정(원·이력)**: Phase 1은 **폴링을 1차 메커니즘**으로 한다. tracker.delivery **Webhook(`registerTrackWebhook`)**은 "검증된 권장 최적화"로 ARCHITECTURE에 설계해두되(콜백 수신 + 24h 재등록 sweep), 도입 시에도 **폴링을 폴백으로 유지**.
**이유(원)**: 과금이 월 고유-운송장 수라 **webhook은 월 쿼터를 절감하지 않는다**(같은 고유 송장). 이득은 신선도↑·cron subrequest↓·헛폴링 제거뿐인데, 24h 재등록·콜백 보안 복잡도가 따른다. 비용이 천장이 아닌 현 상황에선 폴링-first가 더 단순·견고.
**트레이드오프(원)**: 실시간성이 중요해지거나 subrequest 압박이 커지면 webhook 우선 전환. Free에서 webhook 사용 가능 여부 미확인(Open Questions) → 검증 후 활성화.

### ADR-016: 테마 — 시스템 외형 따름, 라이트 기준 설계 (다크 지원)
**결정**: 앱은 **시스템 외형(light/dark)을 따른다**. 디자인 기준선(baseline)은 **라이트**이며 다크는 동등 품질로 지원한다. "다크모드 고정"은 폐기. (`app.json`도 이미 `userInterfaceStyle`를 자동/시스템으로 둔다 — 현재 `light` 고정값을 시스템 추종으로 조정.)
**이유**: 대중 소비자용 택배 추적 앱은 라이트가 표준 기대치(국내 택배사/배송 앱 관행). 다크 전용/다크 디폴트는 이질적이고 신뢰감을 떨어뜨린다. 단 다크 선호 사용자를 위해 다크는 지원해야 한다(원래 MVP 요구의 "다크모드"는 *전용*이 아니라 *지원*으로 재해석).
**트레이드오프**: 라이트/다크 두 팔레트를 다 관리해야 함 → 색을 하드코딩하지 않고 **시맨틱 토큰**으로 추상화(UI_GUIDE). 초기 디자인 노력 ↑.

### ADR-017: 계정 없는 "데이터 삭제" 경로 제공 (스토어 정책 충족)
**결정**: 계정은 없지만(ADR-002) **인앱 "모든 데이터 삭제"**를 설정 화면에 명확히 제공 — `device_id` + 구독 + (orphan) 송장 + 푸시 토큰을 서버에서 폐기(`DELETE /me`). 보조로 개인정보처리방침에 데이터 삭제/문의 경로 명시.
**이유**: Apple 5.1.1(v)와 Google Play User Data 정책은 데이터 수집 앱에 **삭제 경로**를 요구한다(계정 유무와 별개로 데이터 삭제/동의 철회 수단 필요, 설정에 눈에 띄게). 미제공 시 심사 거절·삭제 리스크.
**트레이드오프**: 삭제 후 복구 불가(개인정보 비영속과 일관). 삭제는 즉시·완전(법적 보존 의무 없음). UX상 확인 다이얼로그 필수.

### ADR-018: 알림은 거래성(transactional)만 — 광고성 정보 규제 회피
**결정**: 푸시는 **배송 상태 변화(거래성/정보성) 알림으로 한정**. 마케팅·프로모션 푸시는 Phase 1에서 발송하지 않는다.
**이유**: 정보통신망법상 **영리 목적 광고성 정보**는 사전 수신동의 + `[광고]` 표기 + 야간(21시~익일 08시) 별도 동의가 필요하다. 거래성 알림은 이에 해당하지 않아 규제를 피하고 마찰을 없앤다. Apple/Google도 푸시를 광고 용도로 강요하는 것을 금지한다.
**트레이드오프**: 마케팅 채널로 못 씀(Phase 2에 별도 동의·표기·야간 제한 도입 시 가능). ~~조용시간(PRD)은 과알림 방지 차원이며 법적 야간 제한과는 별개로 운영.~~ **개정(ADR-030, 2026-06-27): 조용시간 폐지** — 거래성 알림은 법적 야간 제한 비대상이고 과알림은 전환 1회·`NOTIFYING_STAGES`·dedupe·음소거로 충분히 억제되므로 **시각 무관 즉시 발송**.

### ADR-019: App Review/심사용 데모 경로 제공
**결정**: 심사자가 **실제 배송 없이** 앱 기능을 검증할 수 있도록 — ① 캔드(canned) 응답을 돌려주는 **데모용 운송장 번호**(또는 데모 모드)와 ② 제출 시 **리뷰 노트**(샘플 번호·예상 동작)를 함께 제공.
**이유**: 추적 앱은 등록 직후 "미등록"만 보일 수 있어 심사자가 핵심 가치(상태 변화→알림)를 못 볼 위험 → minimum functionality(Apple 4.2)·기능 미확인 거절 방지.
**트레이드오프**: 데모 경로를 코드/데이터에 두어야 함(소량). 데모 번호는 실제 폴링을 타지 않도록 분기.

### ADR-020: 송장별 알림 음소거는 per-subscription
**결정**: 송장별 "이 알림만 끄기"는 **구독(subscription) 단위**로 둔다 — `subscriptions.muted`(device_id+shipment_id). 전역(모든 송장)·송장 단위(shipments 행)가 아니다. `PATCH /shipments/:id {muted}` 로 토글하며, 음소거된 구독은 폴링 알림 fan-out의 토큰 소스(`subscriberTokens`)에서 제외돼 **모든 푸시**(단계 전환 + 운영성: 번호 확인·분실 의심)를 받지 않는다.
**이유**: 운송장은 dedupe로 여러 device가 공유(N:1, ADR-007의 소유권은 구독으로 판정)한다. 음소거를 shipments 행에 두면 한 사용자의 끄기가 같은 송장을 구독한 타인에게 새어 나간다. 사용자 멘탈모델("이 택배 알림 끄기")은 단계 전환·운영성을 구분하지 않으므로 음소거는 그 송장의 모든 푸시를 끈다. **단계 추적(폴링·CAS·status_changed_at)은 계속**되고 발송만 빠진다 → 앱을 열면 최신 상태가 보인다.
**ADR-018(거래성 알림만)과의 관계**: 위반 아님. 음소거는 거래성 알림을 **덜 보내는** 방향(수신자 선택)이라 광고성 규제와 무관하다.
**트레이드오프**: 야간 보류 큐(`notification_queue`)는 발송 시점 스냅샷이라 음소거 시점에 이미 적재된 분은 `flushQueue`가 `subscriberTokens`를 거치지 않고 직접 발송한다 → 음소거 토글 시 **해당 기기의 보류분을 함께 정리**(누락 방지). `flushQueue`를 `subscriberTokens` 경유로 리팩터하지 않는다(스냅샷 설계 유지·단순).

### ADR-021: 대시보드 집계는 클라이언트에서 (새 서버 엔드포인트 없음)
**결정**: 대시보드의 요약 수치(진행 중·배송완료·예외 개수 등)는 **앱이 `GET /shipments` 응답(또는 오프라인 캐시)에서 클라이언트로 집계**한다. 전용 집계 엔드포인트(`GET /stats` 등)를 만들지 않는다. 휴지통 개수는 **로컬 휴지통**(ADR-022), 알림 미읽음 개수는 **로컬 읽음 상태**(ADR-023)에서 계산한다.
**이유**: 서버는 SOT, 앱은 호출만(ADR-014)이라 목록을 이미 받아온다. 단계별 카운트는 그 목록의 순수 함수 집계(`dashboardCounts`)로 충분 — 서버 라운드트립·subrequest·새 계약을 늘리지 않는다($0·단순). 오프라인에서도 캐시 목록으로 동일 집계가 가능(오프라인 읽기, ADR-014).
**트레이드오프**: 송장 수가 매우 많아지면 클라이언트 집계 비용이 생기지만 Phase 1 규모(디바이스당 활성 상한 100, ADR-008)에선 무시 가능. 다중 기기 합산 통계가 필요해지면 Phase 2(계정)에서 서버 집계로 전환.
**정정(2026-06-23 — phase 11 A1·A2, 결정은 유지·표면만 축소)**: 구현 후 iOS 스모크 결과 대시보드/필터 표면을 다음과 같이 축소한다(클라이언트 집계라는 본 ADR 핵심은 불변).
- **A1 — 대시보드 4카드**: 카드는 **진행 중·배송완료·휴지통·새 알림** 4개. "확인 필요(예외)"·"오늘 도착" 카드는 **폐기**. **예외는 진행 중에 흡수**(집계는 `stageBucket`이 `완료`가 아니면 전부 진행 중) — `dashboardCounts`는 `{inProgress, completed, trash, unread, amountTeaser}`만 산출(예외·오늘도착 카운트 제거). 카드는 항목별 의미색(진행 중=`accent`/배송완료=`stage.delivered`/새 알림=`stage.inTransit`/휴지통=`stage.exception`, count>0 시).
- **A2 — 택배함 필터 칩 철회(superseded)**: 본 ADR의 "각 카드 탭 → 택배함(해당 **필터 적용**)" 라우팅과 필터 칩(전체/진행 중/임박/완료/예외)을 **철회**한다. 대시보드 카드는 **필터 프리셋 없이** 택배함 탭으로만 진입한다. 택배함에는 **"배송 완료된 항목 감추기" 토글만** 남기고(설정 → 택배함 상단으로 이동), `filterShipments`는 `hideCompleted` 전용(`stageBucket` 함수 자체는 `dashboardCounts`가 계속 사용 — 드리프트 금지). 향후 필터 재도입은 별도 결정으로.

### ADR-022: 삭제 복구(휴지통)는 클라이언트 로컬 소프트 삭제 + 멱등 재등록 복구
**결정**: "삭제 후 30일 복구"는 **기기 로컬 휴지통**으로 구현한다. 삭제 시 ① 서버에 `DELETE /shipments/:id`(구독 해제 — 추적·폴링 즉시 중단)를 그대로 보내고, ② 삭제 직전의 송장 스냅샷(`carrier`·`tracking_no`·`status`·시각들 + 로컬 택배 정보[메모·카테고리·금액])을 **로컬 휴지통 스토어**에 적재한다. **복구 = `POST /shipments(carrier, tracking_no)` 재등록**(멱등 dedupe + 즉시 1회 track) → 현재 상태로 되살아나고, 보관해 둔 택배 정보를 라이브 스토어로 복원한다. **영구 삭제 = 로컬 휴지통 항목 제거**(서버는 이미 구독 해제됨 → 추가 호출 없음). 30일 경과분은 앱 진입/휴지통 열람 시 **로컬에서 자동 영구 삭제**(`now` 주입 순수 함수 `pruneTrash`). 휴지통 키는 ephemeral `shipment.id`가 아니라 **dedupe 키 `(carrier, tracking_no)`**로 둔다(복구 재등록 시 orphan 정리로 행 id가 바뀔 수 있음).
**이유**: 삭제한 송장을 서버가 계속 폴링하면 subrequest·쿼터를 **순수 낭비**(삭제했으니 푸시도 안 보냄)한다. 로컬 휴지통은 서버 스키마·폴링 모델·정리 cron을 건드리지 않고(0 마이그레이션), 기존 **로컬 전용 메모(ADR-024 선행)·비영속 원칙**과 일관된다. 복구를 재등록으로 처리하면 멱등 등록 경로를 재사용해 **별도 복구 API가 불필요**하고, 되살린 송장이 항상 최신 상태로 보인다.
**트레이드오프**: 휴지통은 기기 로컬이라 **앱 삭제·"모든 데이터 삭제" 시 함께 사라진다**(기기 간 복구 불가) → Phase 2 계정 동기화로 해결. 휴지통 체류 중에는 추적이 멈춰 그 기간의 상태 변화 푸시를 못 받는다(삭제 의도와 일치 — 의도된 동작). 동일 `(carrier, tracking_no)`를 사용자가 휴지통과 별개로 수동 재등록하면 휴지통에서 **그 항목을 자동 제거**한다(중복 표시 방지). 대량 복구는 등록 레이트 제한(ADR-008)에 걸릴 수 있어 순차·`429` graceful 처리.
**보강(2026-06-23)**: 삭제 시 info 스냅샷은 `pruneMemos` 보다 **먼저** 기록(유실 방지), 복구 성공 시 info 는 **반환된 shipment.id** 로 귀속, sync 가 서버 목록에 다시 나타난 키를 휴지통에서 reconcile, `pruneTrash` 는 bootstrap·열람 시 + **용량 상한**. 상세 → `ARCHITECTURE.md` "v1.1 설계 보강 ④".

### ADR-023: 알림 기록은 서버측 비영속 상태 로그(SOT) + 로컬 캐시, 읽음은 로컬
**결정**: "받은 푸시 알림 기록"은 **서버가 발송한 모든 알림을 D1 `notifications` 테이블에 기록**(발송 경로에서 1 write)하고 앱이 `GET /notifications`로 받아 본다(서버 SOT, 로컬 캐시 — ADR-014 패턴). 행은 **`device_id`로 키**하고 표시에 필요한 비-PII 필드(`carrier`·끝4자리·`body`·`stage`·`shipment_id`(nullable)·`sent_at`)를 **denormalize**해 자족적으로 저장한다. **읽음 여부(미읽음 배지)는 로컬**(`lastSeenNotificationAt` 타임스탬프) — 서버에 읽음 write 없음. 보존은 cron sweep(예: 90일)+디바이스당 상한.
**이유**: 이 앱의 핵심 가치는 "**앱이 꺼져 있어도** 오는 푸시"라, 사용자가 받는 알림 대부분은 앱 종료 상태에서 도착한다. 클라이언트 로컬 캡처(포그라운드 수신·탭만)는 **가장 흔한 경우(종료 중 수신·미탭)를 누락**해 "지금까지 받은 알림"을 온전히 못 보여준다. 발송하는 서버만이 완전한 기록을 갖는다. 저장 내용은 `{택배사}·…{끝4자리}` + 상태 문구로 **수령인·메모가 없어 비-PII**(ADR-005 위반 아님 — `notification_queue`가 이미 동일 스냅샷을 저장하는 선례). `device_id` 키라 토큰 양도(재설치·복원, `index.ts`의 token steal)에도 **교차 기기 누설이 없다**(token 키인 `push_tickets`·`notification_queue`와 대조).
**트레이드오프**: D1 테이블 1개 + 발송 시 1 write + 보존 cron이 늘지만 무료 티어 내(과금은 월 고유 운송장 수, ADR-006). 음소거(ADR-020)된 구독은 발송에서 제외되므로 **기록에도 안 남는다**(일관). `shipment_id`가 가리키는 송장이 삭제·정리되면 `ON DELETE SET NULL`(+ denormalize 덕에 표시는 유지) — 딥링크만 비활성("정리된 택배"). 재설치로 `device_id` 소실 시 기록도 사라짐(Phase 2 계정).
**보강(2026-06-23)**: 로깅은 **전환 CAS 승리 후 fan-out 시점 1회**(send 재시도 재로깅 ❌). **개정(ADR-030, 2026-06-27)**: 조용시간 폐지로 보류·flush 경로가 사라져 **항상 발송 시점에 로깅**한다(구 "조용시간은 flush 시점" 폐기). `lastSeen` 미설정 시 **첫 fetch 에서 `now` 로 초기화**(기존 기록 미읽음 폭주 방지). 상세 → `ARCHITECTURE.md` "v1.1 설계 보강 ②·⑤".

### ADR-024: 택배 정보(메모·카테고리·금액)는 로컬 전용 + 로컬 스토어 마이그레이션
**결정**: 기존 **로컬 전용 메모**(AsyncStorage)를 **택배 정보 = `{ memo?, category?, amount? }`**로 확장한다. 카테고리·금액도 **서버 미전송·로컬 전용**(메모와 동일 — 마찰 최소·비영속 일관). 모두 **선택 항목**이라 등록 기본 경로는 그대로(번호만으로 등록). 입력은 기존 상세 헤더 연필→모달을 "택배 정보" 모달로 확장(메모+카테고리+금액). 저장 스키마는 `Record<shipmentId, ShipmentInfo>`로 바꾸고, 기존 `Record<shipmentId, string>`(메모 문자열)에서 **1회 로컬 마이그레이션**(`text` → `{ memo: text }`, 멱등·무손실·손상 JSON 안전).
**이유**: tracker.delivery에 상품명·금액·분류가 없어 사용자가 직접 적는다(메모와 동성격). 카테고리·금액은 폴링·알림에 쓰이지 않아 **서버가 알 필요가 없다** → 서버 저장은 익명·비영속 모델에 균열만 낸다. 가계부(카테고리별 금액 합계)는 **Phase 2(계정 도입) 때 서버 동기화로 전환**하고, v1.1은 로컬에 데이터를 모으는 단계로 둔다(대시보드 금액 합계 teaser로 가치 미리보기 가능).
**트레이드오프**: 기기 로컬이라 앱 삭제·"모든 데이터 삭제" 시 사라짐(Phase 2 해결). 마이그레이션은 새 키로 변환 후 구키를 정리 — 구버전 앱으로의 롤백 호환은 보장하지 않는다(단방향, 데이터 저민감).
**보강(2026-06-23)**: 카테고리 **미설정 = 값 없음(칩 없음)** — 별도 "미지정" 의사값 두지 않음. 목록 마지막 `기타`는 실제 catch-all(미설정과 구분). 금액 = **0 이상 정수**(음수·비정수·빈값·상한[≥10^10] 미저장, `parseAmount→number|undefined`). 상세 → `ARCHITECTURE.md` "v1.1 설계 보강 ⑥".

### ADR-025: 기본 시작 화면은 로컬 preference (기본값 = 택배함)
**결정**: 하단 탭에 **대시보드**를 추가(대시보드·택배함·설정 3탭)하고, 설정에 **"시작 화면"**(택배함 / 대시보드) 항목을 둔다. **기본값 = 택배함**. 선택은 테마 preference와 동일하게 **로컬 저장**(서버 무관)이며, 콜드스타트 시 선택된 탭으로 진입한다.
**이유**: 기존 사용자 멘탈모델의 1차 화면은 택배함(목록)이므로 기본은 택배함을 유지(회귀·이질감 방지). 대시보드를 선호하는 사용자는 전환 가능. 시작 화면은 순수 표시 설정이라 서버·계정과 무관(로컬이 옳다).
**트레이드오프**: 콜드스타트 초기 라우팅이 preference 비동기 로드와 경합할 수 있어, preference를 동기 가용한 캐시로 읽거나 게이트 화면에서 1프레임 내 리다이렉트한다(구현 시 SDK 56 라우팅 문서 확인 — `app/AGENTS.md`). 미설정/로드 실패는 택배함 폴백.
**보강(2026-06-23)**: **콜드스타트 라우팅 우선순위 = ① 알림 탭 딥링크(최우선) → ② 시작 화면 preference → ③ 택배함 폴백**. 알림으로 앱이 켜지면 해당 상세가 시작 화면 설정을 이긴다. 상세 → `ARCHITECTURE.md` "v1.1 설계 보강 ③".

### ADR-026: 택배사 자동선택은 후보가 모호하면 안 함 (오선택 방지)
**결정**: 운송장 등록 시 택배사 추정(`estimateCarriers`, 자릿수 휴리스틱)이 **후보를 2개 이상** 반환하면 1순위를 **자동 선택하지 않는다**. 택배사 필드는 "택배사를 선택하세요" 미선택 상태로 두고, 사용자가 드롭다운(이미 존재하는 `CarrierList`)에서 **명시적으로 고른다**(추천 순서·"추천" 태그는 유지). 후보가 **정확히 1개**로 확실할 때만 자동 채움. 등록 버튼은 택배사 미선택이면 비활성.
**이유**: 자릿수 기반 추정은 같은 길이에 여러 택배사가 매핑돼 1순위 자동 채움이 자주 틀린다(실측 발견). 틀린 자동완성은 등록 후 폴링이 엉뚱한 택배사로 가 "미등록"·오추적을 부른다. 자동완성을 끊고 명시 선택을 받으면 마찰은 탭 1회로 최소이면서 정확도가 오른다(마찰 최소 원칙 위배 아님 — 번호만으로 등록하는 흐름은 유지하되 택배사만 확인받는다).
**트레이드오프**: 후보가 거의 항상 2개 이상이라 사실상 매 등록에서 택배사 1탭 선택이 필요해진다(기존 "자동 채움" 대비 탭 1회 증가). 휴리스틱이 정교해져 단일 후보를 자주 낼 수 있게 되면 자동 채움 빈도가 자연히 는다(룰 개선은 별도 결정). 상세 수정(ADR-027)의 택배사 선택도 같은 컴포넌트·정책을 재사용한다.

### ADR-027: 운송장 식별자(택배사·번호) 수정은 클라이언트 재등록 (서버 PATCH 아님)
**결정**: 상세 화면 "수정"에서 **택배사·운송장번호를 바꿀 수 있게** 한다. 단 이는 추적 식별자 변경이라 **서버 행을 직접 수정하지 않고** 클라이언트가 **재등록**으로 처리한다: ① `POST /shipments(새 carrier, 새 tracking_no)` 등록(**먼저** — 실패 시 기존 유지) → ② 로컬 택배 정보(메모·카테고리·금액)를 **old `shipment.id` → 반환된 new `shipment.id` 로 이관** → ③ `DELETE /shipments/{old_id}`(기존 구독 해제) → ④ 새 상세로 이동. 새 `(carrier,tracking_no)`가 기존과 같으면 no-op. 새 조합이 **이미 추적 중**이면(dedupe hit) "이미 추적 중인 운송장" 안내 후 그 상세로 이동(중복 구독 생성 안 함). 서버에 **새 수정 엔드포인트를 만들지 않는다**(기존 `POST`·`DELETE` 재사용).
**이유**: `shipments` 행은 `UNIQUE(carrier,tracking_no)`로 dedupe되고 **여러 기기가 공유**(N:1 구독)한다. 서버에서 행의 carrier/tracking_no를 PATCH하면 **같은 운송장을 추적하던 다른 기기에도 변경이 전파**되는 부작용 + `UNIQUE` 충돌 처리가 필요하다. 재등록은 그 부작용이 없고(이 기기 구독만 새 운송장으로 옮김), 휴지통 복구(ADR-022)가 이미 쓰는 **재등록 + info 이관** 패턴을 그대로 재사용해 새 백엔드·배포·스모크가 불필요하다. 식별자가 틀렸다는 건 "사실 다른 택배"라는 뜻이라 새로 추적을 시작하는 게 의미상으로도 옳다.
**트레이드오프**: 수정 후 **새 번호로 폴링을 새로 시작**해 이전 번호의 타임라인·폴링 이력은 이어지지 않는다(잘못된 번호였으니 의도와 일치). info는 `shipment.id`로 이관되어 보존되나, 휴지통에 옛 번호 스냅샷이 남아 있던 경우는 자동 갱신되지 않는다(수정은 휴지통과 무관 경로). 순서를 **등록-먼저·삭제-나중**으로 둬 중간 실패 시 기존 구독을 잃지 않게 한다(휴지통 삭제의 "스냅샷 먼저" 규칙과 동형). 상세 → `ARCHITECTURE.md` "v1.1 추가 — 식별자 수정".

### ADR-028: webhook-first + cron 유지보수/폴백 (supersedes ADR-015)
**결정**: 신선도 1차 메커니즘을 **tracker.delivery webhook**으로 전환한다. `POST /shipments` 등록 시 `registerTrackWebhook(carrierId, trackingNumber, callbackUrl, expirationTime)`로 **active·비종료 송장당 webhook 1개** 등록(dedupe — 여러 device 구독이어도 1개; **dedupe-hit·이미 등록분(`webhook_expires_at` 존재)은 재등록 없이 만료만 연장 → 멱등**; 종료(배송완료)는 등록 안 함; **미등록(이벤트 0)은 등록 보류 → 폴링이 첫 이벤트 감지 후 등록(폴링→webhook 승급); 운영 D1 기존 추적분(`webhook_expires_at` NULL active)은 cron webhook 등록 sweep 이 due 무관 매 fire 등록해 즉시 마이그레이션(소수라 첫 fire 일괄) — 등록 실패 재시도도 겸함**). 변화 시 콜백 → 재조회 → 기존 정규화·CAS·푸시 다운스트림 재사용. **cron(15~30분)은 유지보수 + 저빈도 폴백**으로 남긴다: ① webhook 등록/재등록 sweep — NULL active 등록가능분 **즉시 등록**(기존 풀 마이그레이션·등록 실패 재시도, due 무관) + 만료 임박분(<24h) **재등록**(TTL 48h) ② lifecycle 독립 sweep(미등록7일·분실30일 — 폴링 루프에서 분리) ③ **조건부 폴백 재폴링** — webhook 등록분(`webhook_expires_at` 있음)은 ~12h 안전망, **미등록분(등록 실패·미등록 단계·1000 초과; `webhook_expires_at` NULL)은 기존 적응형 간격 그대로**(이 송장은 폴링이 1차) ④ 기존 receipt/flush/보존/rate_limits 정리.
**이유**: Free webhook 검증(1000 동시·무료 확장)으로 ADR-015의 활성화 조건 충족. webhook은 월 쿼터를 줄이진 않지만(고유-번호 과금) **신선도(실시간 알림)·cron subrequest·헛폴링 제거**라는 본 앱의 핵심 가치("상태 변화 시 푸시")에 직결된다. 콜백도 페이로드를 불신하고 `track`을 재조회하므로 폴링과 **동일한 정규화·CAS·멱등 다운스트림**을 공유 — 신규 표면은 등록·수신·재등록 sweep뿐이고 알림 로직은 불변.
**트레이드오프**: webhook 등록/재등록(24h)·콜백 보안(ADR-029) 복잡도가 는다. **활성 webhook 1000 동시 한도** — 초과분은 등록 실패로 폴백 폴링이 흡수(무료 확장 요청 전까지 안전). webhook이 "변화 없음"을 감지 못하므로 **lifecycle 판정을 폴링에서 독립 sweep으로 분리 필수**(안 하면 미등록·분실 만료가 누락). 폴백 폴링 유지로 단순성은 ADR-015 원안보다 낮지만, 폴링-only가 이미 검증·운영 중이라 폴백은 **기존 코드 재사용**(추가 위험 작음). **슬롯 위생**: DELETE(마지막 구독)·비활성(`active=0`) 시 재등록 sweep에서 빠져 webhook이 **≤48h 자연 만료로 1000 슬롯 회수**(deregister API 있으면 즉시 — 스모크). 그동안 잔여/위조 콜백은 페이로드 불신(D1 active 아님)으로 무해.

### ADR-029: webhook 콜백 인증 — 시크릿 경로 + 페이로드 불신 (인증 없는 콜백 보완)
**결정**: tracker.delivery 콜백은 발신자 인증 수단이 없으므로 **3중 방어**로 막는다. ① **추측 불가 시크릿을 경로 세그먼트**에 박는다(`/webhooks/track/<secret>`, `WEBHOOK_CALLBACK_SECRET`) — 쿼리스트링(`?k=`)은 액세스 로그·Referer 누출 위험이라 경로 사용. 불일치는 조용히 `401`. ② **페이로드 불신** — 콜백의 `{carrierId, trackingNumber}`를 신뢰하지 않고, **우리 D1에 active 송장으로 존재할 때만** `track` 재조회·처리(없으면 무시). 위조 콜백이 임의 번호로 quota를 태우는 것을 차단. ③ **송장별 신선도 throttle** — 콜백 수신 시 `last_polled_at` 선점 갱신(ADR-012)으로 같은 송장의 동시·연속 콜백을 dedupe하고, 직전 폴링이 너무 최근(예 <60s)이면 재조회를 건너뛴다. **IP 기준 rate limit은 쓰지 않는다** — 콜백은 tracker.delivery의 소수 고정 IP에서 와 IP throttle이 정상 콜백을 한꺼번에 막는다(거짓양성). tracker.delivery가 **서명(HMAC)을 제공하면** `WEBHOOK_SIGNING_SECRET`로 검증 추가(스모크). **응답 모델**: 시크릿 검증·active 송장 확인은 동기(빠른 D1 읽기) → 시크릿 불일치 `401`, 미존재/비active `202`(무시), 수락 시 `202` + `ctx.waitUntil`로 track 재조회·CAS·푸시. **track 실패는 `202` 이후라 tracker 재시도가 아니라 폴백 폴링이 흡수**. CAS로 중복 콜백 멱등.
**이유**: 익명·무인증 공개 콜백이라 우리가 막아야 한다. 시크릿 경로가 1차 게이트, 페이로드 불신이 핵심 — 어차피 `track`을 재조회하므로(ADR-011·콜백 페이로드엔 상태 없음) "D1 active 송장만 처리"가 자연스럽고, 위조 콜백 최악의 영향은 **존재하는 송장에 대한 헛 재조회 1회**(rate limit으로 상한)뿐. 알림은 CAS+`shouldNotify`라 **푸시 스팸 불가**(단계 전환만 발송).
**트레이드오프**: 시크릿 경로가 유출되면(로그·중간자) 무력화되나 페이로드 불신·신선도 throttle이 잔여 방어. 서명 검증은 tracker.delivery 지원 시에만 가능(미지원이면 경로+불신으로 충분 판단). 시크릿 경로는 **로그 금지**(device_id와 동급)이나 **Cloudflare 엣지 로그/`wrangler tail`에 URL로 남을 수 있어**(우리 코드 로그만 통제 가능) → **페이로드 불신이 실질 1차 방어**, 경로는 보조 게이트로 본다. 시크릿 비교는 **상수시간(timing-safe)**.

### ADR-030: 알림 즉시 발송(조용시간 폐지) + 이동중 최초 1회 알림 (2026-06-27)
**결정**: ① **조용시간(야간 보류) 폐지** — 모든 거래성 단계 전환 푸시를 **시각과 무관하게 즉시** 발송한다. 야간 보류 큐(`notification_queue`)·아침 묶음(`flushQueue`)·`isQuietHours`/`isUrgentStage` 제거. ② **`이동중`을 알림 단계(`NOTIFYING_STAGES`)에 추가** — CAS(전환 1회·`prev !== next`)로 **첫 진입 시 1회만** 발송("상품이 이동을 시작했어요" 무드), 이후 터미널 입·출고(모두 같은 `이동중` 단계)는 무발송.
**이유**: ① 야간 전송 제한(21–08)은 **광고성 정보 전용**(정보통신망법) — 배송 알림은 거래성이라 비대상이고, 사용자가 옵트인한 정보성 알림이라 즉시성이 가치다. 과알림은 이미 **단계 전환 1회·`NOTIFYING_STAGES` 한정·dedupe·음소거(ADR-020)**로 억제돼 조용시간은 한 겹 중복(긴급 배송완료·예외는 이미 즉시였고, 등록·집화·배송출발도 송장당 평생 몇 건뿐). ② 사용자가 "상품이 움직이기 시작했다"를 알고 싶어 함. `이동중`은 터미널마다 이벤트가 많아 매번 알리면 과알림 → **CAS 멱등으로 최초 1회**만 자연 충족(추가 상태 불필요).
**트레이드오프**: 알림 로깅이 **항상 발송 시점 1회**로 단순화(ADR-023 flush-시점 로깅 경로 제거). `notification_queue` 테이블은 **미사용으로 남김**(DROP 안 함 — 운영 D1 드리프트·hook, P-3). 이동중 추가로 송장당 푸시 1건↑(사용자 요청·허용). 음소거(ADR-020)·dedupe는 그대로.
**supersedes**: ADR-018 트레이드오프의 "조용시간 운영" · ADR-023 "조용시간 = flush 시점 로깅".

### ADR-031: 미등록 폴링 시간대 윈도 (KST, 2026-06-27)
**결정**: webhook 등록 전(`미등록`) 송장의 폴백 폴링 간격을 **KST 시간대로 조절**한다(`unregisteredInterval`). webhook-first 전환 후 대부분 단계는 콜백·~12h 백스톱이 처리하지만, `미등록`은 webhook 불가(tracker가 번호를 아직 모름)라 **폴링이 유일한 첫-이벤트 감지 수단**이다.
- **06:00–18:00** → 15분 (cron 15분 주기 바닥, now=scheduledTime 그리드 정렬이라 매 fire 폴링)
- **18:00–21:00** → 1시간
- **21:00–06:00** → **재폴링 skip**, 단 **한 번도 안 본 송장(`last_polled_at` NULL)은 1회 허용**(밤에 등록한 택배의 첫 확인) — 이후 06:00까지 재폴링 안 함. (isDue 가 interval=null·미등록·미폴링일 때만 통과.)
**이유**: 첫 이벤트(집화·등록)는 주로 주간에 찍힌다. 6h 단일이던 종전은 첫 알림이 최대 6h 지연 → 주간 15분으로 최소화. 야간엔 택배사 업데이트가 거의 없어 재폴링은 헛수고라 skip하되, 밤에 막 등록한 송장은 낮에 이미 집화됐을 수 있어 1회는 확인. 월 쿼터는 고유-번호 과금이라 빈도 무관, 부담은 fire당 subrequest(≤50)뿐이라 안전.
**적용 범위**: `미등록`에만. 다른 단계는 24/7 이벤트(이동중 허브)·webhook 콜백이 처리하므로 시간대 무관(적응형·12h 백스톱 유지). cron `MIN_POLL_INTERVAL_MS` SQL 1차 바닥을 60분→15분으로 낮춰 주간 15분 허용(정밀 판정은 isDue).
**트레이드오프**: 미등록 폴링이 시간 의존이 돼 테스트에 now 주입 필수(이미 그러함). 밤 등록 송장은 1회 후 06:00까지 미확인(의도 — 야간 헛폴링 방지).
