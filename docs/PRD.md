# PRD: unboxing (택배 추적 앱)

> 기획 문서. 아키텍처/구현은 `ARCHITECTURE.md`, 결정 근거는 `ADR.md`, 화면은 `UI_GUIDE.md`.

## 배경 / 문제

국내 사용자는 여러 택배를 동시에 받지만, 택배사 앱/웹은 제각각이고 "직접 들어가서 조회"해야 한다. 알림은 택배사마다 다르거나 없다. 핵심 페인: **상태가 바뀐 걸 사용자가 먼저 챙겨야 한다.**

- **핵심 가치**: "안 봐도 오는 알림" — 운송장만 등록해두면 **앱이 꺼져 있어도** 서버가 추적하다 상태 변화 시 푸시.
- 비기술 일반 대중도 **회원가입·설정 없이** 30초 안에 첫 송장을 등록할 수 있어야 한다.

## 목표

운송장을 등록하면 백그라운드(서버)에서 상태를 추적하고, 표준화된 단계가 바뀔 때만 정확히 1회 푸시한다. 운영비 0원으로.

## 사용자

- **주 사용자**: 국내 택배를 받는 일반 대중. 로그인·키 입력 등 마찰을 극도로 기피.
- **부 사용자**: 여러 건을 동시에 받는 사람(쇼핑 잦은 사용자), 가족 택배를 대신 챙기는 사람.
- 전제: 한국어, 국내 택배사(Phase 1).

## 사용자 스토리

- 사용자는 운송장 번호를 **붙여넣거나 입력**해 추가하고 싶다 — 택배사는 앱이 추정해주길 바란다.
- 사용자는 **상태가 바뀔 때만** 알림 받고 싶다(이동 중 매 스캔마다 말고).
- 사용자는 앱을 **켜지 않아도** 배송출발/완료를 알고 싶다.
- 사용자는 **로그인 없이** 바로 쓰고 싶다.
- 사용자는 받은 택배를 **목록에서 지우고** 싶다.

## 핵심 플로우

1. **등록**: 앱 실행 → (클립보드에 운송장-형태 번호면 추가 제안) → 번호 입력 → 택배사 자동추정/확인 → 등록 → 즉시 1회 조회 → 목록에 표시.
2. **추적/알림**: 서버 cron이 주기 폴링 → 단계 전환 시 푸시 → 사용자가 알림 탭 → 상세 타임라인(딥링크).
3. **수령**: `배송완료` 감지 → 완료 알림 → (정책상) 목록에서 정리.
4. **미지원 택배사**: 자동 추적 대신 택배사 조회 페이지 **딥링크** + "자동 추적 미지원" 안내.
5. **권한 거부**: 등록·인앱 조회는 가능, 알림 불가 명시 + 설정 유도.
6. **미등록/오타 번호**: 활성 폴링(6h), 7일 미수신 시 "번호 확인" 안내 후 비활성.

## 핵심 기능 (Phase 1 / MVP)

1. 운송장 등록 — 수동 입력 + 클립보드 감지(앱 열 때 운송장-형태 번호면 추가 제안)
2. 택배사 자동인식 + 확인 — 번호 형식으로 후보 로컬 추정 → **후보 1개면 자동 채움·여러 개면 자동선택 없이 드롭다운 명시 선택**(오선택 방지, ADR-026), 추정 불가면 수동 드롭다운
3. 송장 목록 / 상세 타임라인(실시간 조회)
4. 서버리스 cron 폴링 + 표준 단계 정규화
5. 푸시 알림 — 등록·집화·배송출발·배송완료·예외 (`이동중`·`기타`·`미등록`은 무알림)
6. 수동 삭제 · 당겨서 새로고침
7. 푸시 권한 온보딩
8. 라이트/다크 테마 — 시스템 외형 따름, **라이트 기준**(다크 전용/디폴트 아님, → ADR-016)
9. 설정 화면 — 알림 토글·테마·개인정보처리방침 링크·**모든 데이터 삭제**·앱 버전
10. 개인정보처리방침 (앱스토어 필수)

## 알림 정책

- **단계 전환에만** 1회(멱등). 단계별 문구 예시:
  - `등록`: "{택배사} 접수 확인 — {별칭/끝4자리}"
  - `집화`: "택배사가 상품을 수거했어요"
  - `이동중`: "상품이 이동을 시작했어요" (첫 진입 1회만 — ADR-030)
  - `배송출발`: "오늘 도착 예정 — 배송이 시작됐어요"
  - `배송완료`: "배송 완료 ✓"
  - `예외`: "배송에 문제가 있어요(지연/반송) — 확인이 필요해요"
- **즉시 발송(조용시간 폐지 · ADR-030)**: 모든 단계 전환 푸시를 **시각 무관 즉시** 발송(거래성=야간 제한 비대상). 과알림은 전환 1회·dedupe·음소거로 억제. (구 조용시간 야간 보류·아침 묶음 폐기.)
- **이동중 최초 1회(ADR-030)**: `이동중` 첫 진입 시 1회만, 이후 터미널 입·출고는 무발송(CAS 멱등).
- **Android notification channel** 분리(배송 상태 채널). 알림 탭 → 해당 **상세로 딥링크**(payload에 shipment_id).

## 권한 & 온보딩

- 시스템 권한 팝업 전에 **사전 안내(pre-permission priming)**: "상태가 바뀌면 알려드리려면 알림이 필요해요". 첫 등록 직후 등 가치가 분명한 시점에 요청.
- **Android 13+ `POST_NOTIFICATIONS`** 런타임 권한 — priming 후 요청. 거부 시: 앱 사용 계속 가능(알림만 비활성) + 설정으로 가는 안내. 강제·반복 팝업 금지.
- **클립보드 정책**: iOS는 클립보드를 읽을 때 시스템 배너가 뜬다 → 앱 열자마자 자동 읽기 대신, 사용자가 등록 화면에 들어왔을 때 등 **명시적 가치 시점에만** 읽고 **제안만**(자동 등록 금지). 권한 사용 설명 문자열에 사유 명시.

## UX 세부

- **첫 실행**: 빈 목록 + 한 줄 가치 제안("운송장만 넣어두면 상태가 바뀔 때 알려드려요") + 첫 등록 CTA. 강제 튜토리얼 없음.
- **마이크로카피 톤**: 친근한 한국어. 사용자에게 **에러 코드/기술 용어 노출 금지**(내부 code는 로그만). 예: 실패 시 "잠시 후 다시 시도해 주세요".
- **상대 시간**: "방금 업데이트", "2시간 전" 등 KST 기준 상대 표기 + 절대 시각 보조.
- **알림 그룹화**: 여러 송장 알림은 묶음/요약으로(과알림 방지, 조용시간과 결합).
- **삭제 확인**: 스와이프 2단계/버튼 탭으로 삭제 시 **확인 다이얼로그**("삭제할까요?")로 실수 방지(단건·일괄 통일, Undo 토스트는 폐기 — v1.1, 회귀 금지). 30일 복구는 휴지통(아래 v1.1). 스와이프 2단계 도달 시 **햅틱** 피드백.
- **마지막 업데이트 표시 + 당겨서 새로고침**: 목록/상세에 신선도 표기.
- **목록 정렬**: 진행 중 우선, 임박(`배송출발`)·`예외` 강조. (완료는 정리되므로 잔여는 진행 건 위주 — ADR-005.)
- **택배사 인식**: 카드에 택배사명/색으로 빠른 식별(로고 사용 시 상표 가이드 준수 → UI_GUIDE).
- **데이터 삭제 UX**: 설정의 "모든 데이터 삭제"는 확인 다이얼로그 + 결과 안내(복구 불가 명시).

## 비기능 요구 (NFR)

- **성능**: 앱 콜드스타트 후 목록 표시 빠르게(서버 응답 + 로컬 캐시 우선 렌더). track 상세는 upstream 권장 timeout 15s 내.
- **신뢰성**: 알림 **누락 0 지향**(멱등 + receipt 기반 토큰 위생). 중복 발송 금지.
- **오프라인**: 마지막 상태를 로컬 캐시로 읽기 가능(변경은 온라인).
- **배터리/데이터**: 추적은 서버가 하므로 앱은 가벼움(백그라운드 fetch 미사용).
- **접근성**: 색만으로 상태 구분 금지(아이콘+텍스트), 대비 WCAG AA, 다이내믹 타입, 스크린리더 라벨(→ UI_GUIDE.md).
- **운영**: 무인 운영 지향(단 tracker.delivery Free 자격증명 21일 재발급은 예외 — ADR-013).

## 지원 범위

- 플랫폼: iOS, Android (Expo 단일 코드베이스).
- 최소 OS: Expo SDK 56 지원 범위(구체 버전은 빌드 시 확정).
- 택배사: 국내 주요 + tracker.delivery 지원 목록(미지원은 딥링크 폴백).
- 언어: 한국어.

## 스토어 정책 & 컴플라이언스

> 최종 산출물은 앱 배포다. 아래는 제출/심사 통과 필수 사항. 버전 민감 항목은 **제출 시점 최신 정책 재확인**.

### 공통
- **개인정보처리방침(한글) URL** 필수(앱·스토어 양쪽): 수집항목(운송장·푸시토큰), 목적, **제3자 제공**(tracker.delivery로 운송장 전송 + ToS상 추적데이터 공유), **국외이전**(Cloudflare), 보유·삭제, 문의처.
- **데이터 삭제 경로**: 계정이 없어도 인앱 "모든 데이터 삭제"를 설정에 명확히 제공(→ ADR-017, `DELETE /me`).
- **거래성 알림만**: 푸시는 배송 상태에 한정(광고성 아님 → 규제·정책 회피, ADR-018).
- **데모/리뷰 경로**: 심사자가 실제 배송 없이 검증 가능하도록 데모 운송장 + 리뷰 노트(→ ADR-019).
- **전송 암호화**: 모든 통신 HTTPS(평문 없음) → Apple 수출규정 표준 면제 선언.

### Apple App Store
- **App Privacy(Nutrition Labels)**: 운송장/푸시토큰 수집 신고, 수령인 이름·주소·연락처는 **저장 안 함**(ADR-005), ATT/광고식별자 미사용.
- **데이터 삭제(5.1.1(v))**: 데이터 삭제/동의 철회 수단을 설정에 눈에 띄게. (계정 미사용이라 "계정 삭제"는 N/A지만 데이터 삭제 경로 필요.)
- **Privacy Manifest**(`PrivacyInfo.xcprivacy`) + required-reason API 신고: Expo/RN 및 사용 SDK 매니페스트 포함(클립보드 등 reason 코드).
- **푸시 정책**: 앱 사용에 푸시 **강제 금지**(거부해도 동작), 광고/프로모션 사용 금지.
- **Minimum functionality(4.2)**: 단순 웹 래퍼가 아닌 실사용 가치(데모 경로로 입증).
- **연령 등급**: 4+ 예상.

### Google Play
- **Data Safety form**: 수집/공유 데이터 신고 — **제3자 SDK(Expo Push·Cloudflare 백엔드)도 반드시 포함**. 데이터 삭제 질문에 "삭제 메커니즘 제공" 표기.
- **데이터 삭제**: 인앱 삭제 경로(+선택적으로 웹 삭제 요청 URL).
- **`POST_NOTIFICATIONS`(Android 13+)**: 런타임 권한 — priming 후 요청, 거부 graceful.
- **권한 최소화**: 불필요 권한 미요청. **포그라운드 서비스 미사용**(추적은 서버 cron) → FGS 정책 부담 없음.
- **Target API level**: Play 정책상 최신 근접 target SDK 요구 → 빌드 시점 기준 충족(Expo SDK 56 빌드로 대응).
- 개인정보처리방침 URL 필수.

### 한국 법규
- **개인정보보호법(PIPA)**: 한글 개인정보처리방침, 수집·이용 동의, **제3자 제공**(tracker.delivery)·**국외이전**(Cloudflare) 고지/동의.
- **정보통신망법**: 배송 알림은 거래성/정보성이라 광고성 정보 규제 비대상. (향후 마케팅 푸시 도입 시 사전 수신동의 + `[광고]` 표기 + 야간 21~08시 별도 동의 — ADR-018.)
- **위치정보법**: 사용자 위치 미수집(택배 허브명은 사용자 위치 아님) → 위치정보사업 신고 대상 아님.

## MVP 완료 기준 (Definition of Done)

- 등록(수동+클립보드 제안) → 자동인식/확인(미지원 딥링크) 동작.
- 목록/상세(실시간 타임라인), 삭제, 당겨서 새로고침.
- 서버 cron 폴링 + 정규화 + **단계 전환 멱등 푸시**(등록·집화·배송출발·배송완료·예외).
- 푸시 권한 온보딩(거부 graceful), **라이트/다크 테마(시스템 따름, 라이트 기준)**, 설정 화면.
- **스토어 제출 준비**: 개인정보처리방침 URL, 인앱 데이터 삭제(`DELETE /me`), App Privacy/Data Safety 신고, Privacy Manifest, 데모/리뷰 노트.
- 핵심 순수 로직 테스트 통과(`npm run verify` green).

- 다국어

## v1.1 마이너 업데이트 (대시보드·휴지통·알림 기록·택배 정보)

> Phase 1(MVP) 출시 이후의 마이너 기능 묶음. **기존 제약(익명·로그인 없음·운영비 $0·개인정보 비영속)을 그대로 유지**한다. 결정 근거는 `ADR.md` ADR-021~025, 데이터/흐름은 `ARCHITECTURE.md` "v1.1 …", 화면은 `UI_GUIDE.md`.

### 배경 / 동기
택배가 쌓이면 "지금 몇 개가 오고 있고, 뭐가 끝났는지" 한눈에 보고 싶다. 실수로 지운 택배를 되살리고 싶고, 받은 알림을 다시 보고 싶다. 또 "이게 무슨 택배였는지"를 메모뿐 아니라 **분류·금액**으로 적어두면 나중에 가계부로 이어진다. v1.1은 이 네 가지 + 마찰을 줄이는 부속 기능을 더한다.

### 추가 사용자 스토리
- 사용자는 **대시보드**에서 진행 중·완료·예외 개수와 휴지통·새 알림을 한눈에 보고, 각 항목으로 바로 이동하고 싶다.
- 사용자는 택배를 **지워도 30일 안에는 되살릴 수** 있고, 필요하면 **완전히 삭제**하고 싶다.
- 사용자는 **지금까지 받은 푸시 알림 목록**을 보고, 탭하면 해당 택배 상세로 가고 싶다.
- 사용자는 택배에 **카테고리와 금액**을 (선택적으로) 적어두고 싶다(나중에 분류별 지출을 보기 위해).
- 사용자는 앱을 켰을 때 **택배함 또는 대시보드 중 원하는 화면**으로 시작하고 싶다.
- 사용자는 택배함에서 **완료된 항목을 숨기고** 싶다.

### 기능 명세

**1. 대시보드(요약 화면)** — 하단 탭에 추가(기본은 택배함이 첫 화면, ADR-025).
- 요약 카드 **4개**(2026-06-23 A1 정정 — 6→4 축소): **진행 중**(배송완료가 아닌 전체 — `예외`도 진행 중에 **흡수**), **배송완료**, **휴지통**(복구 가능 건수), **새 알림**(미읽음 수). "확인 필요(예외)"·"오늘 도착" 카드는 **폐기**(대시보드는 예외·오늘도착을 별도 집계하지 않는다).
- 각 카드는 count>0 일 때 **의미색**(진행 중=`accent`/배송완료=`stage.delivered`/새 알림=`stage.inTransit`/휴지통=`stage.exception`), 0이면 중립(`text/secondary`) — 색 단독 금지(아이콘+라벨 동반).
- 각 카드 탭 → 라우팅: 진행 중·배송완료 → **택배함 탭**(필터 프리셋 없음 — A2 로 필터 제거), 휴지통 → 휴지통 화면, 새 알림 → 알림 화면.
- 집계는 **클라이언트**에서 목록(또는 오프라인 캐시)으로 계산(ADR-021). 빈/오프라인 상태 처리.
- (제안) **이번 달 등록 금액 합계** teaser — 로컬 금액 데이터 합(가계부 예고). 범위 = **KST 기준 createdAt 이 이번 달**인 건, **휴지통 제외**. 금액 미입력 건은 합산 제외 + "일부만 입력됨" 안내.

**2. 휴지통(삭제·복구 관리)** — 30일 복구 + 영구 삭제(ADR-022, **기기 로컬**).
- 택배 삭제 시 서버에선 즉시 구독 해제(추적 중단)하되, 삭제 직전 스냅샷(+택배 정보)을 **로컬 휴지통**에 보관.
- 휴지통 화면: 삭제한 택배 목록 + **"N일 후 영구 삭제"** 표기. **복구**(재등록으로 현재 상태 되살림) / **영구 삭제**(즉시 제거). 일괄 복구·일괄 영구 삭제 제공.
- 30일 경과분은 자동 영구 삭제(로컬). "모든 데이터 삭제" 시 휴지통도 폐기.
- 삭제 흐름과 계층 분리: 삭제 시 **확인 다이얼로그**(즉시 실수 방지), 휴지통은 삭제 반영 후 **30일 복구**(되돌리기). 확인 통과 시 서버 구독 해제 + 휴지통 적재.

**3. 알림 기록** — 받은 푸시 목록(ADR-023, **서버 로그 + 로컬 캐시**).
- 서버가 발송한 모든 알림(상태 메시지)을 기록 → 앱이 받아 시간 역순 목록으로 표시(택배사 한글명·상태 문구·상대시간). **앱이 꺼져 있을 때 받은 알림까지** 포함.
- 항목 탭 → 해당 택배 상세 딥링크. 삭제·정리된 택배면 "정리된 택배" 안내(딥링크 비활성).
- **헤더 알림 종 아이콘** + 미읽음 배지(택배함·대시보드 헤더). 알림 화면 열람 시 읽음 처리(로컬). 빈 상태.

**4. 택배 정보(메모·카테고리·금액)** — 메모 입력을 "택배 정보" 입력으로 확장(ADR-024, **로컬 전용**).
- 상세 헤더 **택배 정보 아이콘**(2026-06-23 #2 — 연필 "수정"과 별개의 `FileText` 글리프) → "택배 정보" 모달: **메모 + 카테고리(선택) + 금액(선택, ₩)**. 모두 선택 항목 — 등록 기본 경로(번호만)는 불변(마찰 최소). *(택배사·번호 편집은 연필 "수정" → 아래 "v1.1 추가" #4.)*
- 카테고리: 고정 목록(식품/생활용품/의류·패션/뷰티/전자·디지털/도서·문구/가구·인테리어/유아·반려/기타). **선택 안 함이 기본(미설정)** — `기타`는 실제 catch-all이지 미설정이 아니다. 금액: 0 이상 정수(원).
- (제안) 카드에 **카테고리 칩**을 작게 표시(빠른 스캔, 메모 표시 규칙은 불변).
- 가계부(분류별 지출)는 Phase 2 서버 집계로.

**5. 기본 시작 화면 설정**(ADR-025) — 설정에 "시작 화면"(택배함/대시보드), 기본 택배함. 로컬 저장.

**6. 택배함 완료 숨기기**(2026-06-23 A2 정정 — 필터 칩 폐기) — 진행 상태 **필터 칩(전체/진행 중/임박/완료/예외)은 폐기**(보류). **"배송 완료된 항목 감추기" 토글**만 남기고, 위치를 **설정 → 택배함 상단**으로 이동(설정 "표시"엔 시작 화면 라디오만 유지).
- 토글은 클라이언트 순수 필터(`filterShipments(list, { hideCompleted })`, 서버 무관) → 정렬(기존 `sortShipments`) 전 적용. 송장이 있고 비선택 모드일 때만 노출. 지속 설정(로컬 `unboxing.list_filter`). 멀티선택과 공존.

**7. 부속(당연 포함)** — 헤더 알림 종+라우팅, 대시보드→각 화면 라우팅(위에 포함).

### 추가 제안 기능 (item 8)
- **헤더 알림 종 + 미읽음 배지**(위 3에 포함).
- **대시보드 금액 합계 teaser**(위 1, 가계부 예고).
- **휴지통 일괄 복구/일괄 영구 삭제**(위 2).
- **알림 기록 → 상세 딥링크**(위 3).
- **카드 카테고리 칩**(위 4).
- **푸시 발송 문구의 택배사 한글명**(이슈 `#9`) — 알림 기록 표시 일관성을 위해 함께 처리 권장.

**신규 제안 (2026-06-23 보강 — 스코프 표시):**
- **(권장·소) "오늘 도착 예정"** — 대시보드 카드 + 택배함 빠른 필터. `배송출발` 중 KST 당일 건(기존 `buildMessage` 의 KST 당일 판정 로직 재사용). 가장 행동 유발적 정보라 가치 큼·구현 작음. *(2026-06-23 A1·A2 정정: 대시보드 '오늘 도착' 카드는 4카드 축소로 **폐기**, 택배함 빠른 필터도 필터 제거로 **폐기**. 향후 재도입 검토 후보로만 남김.)*
- **(권장·소) 알림 "모두 읽음" + 날짜 그룹 헤더** — 알림 기록 화면. `lastSeen` 한 번에 갱신 + 시각 가독성. 거의 무비용.
- **(권장·소) 알림 → 휴지통 복구 딥링크** — 정리된 택배 알림 탭 시 그 `(carrier,tracking_no)` 가 로컬 휴지통에 있으면 "복구하기"를 제안(404 안내 대신). ADR-022·023 연계, 데이터 이미 있음.
- **(선택) 대시보드 카테고리별 지출 미리보기** — 이번 달 상위 카테고리 합계(가계부 Phase 2 예고 강화). 금액 입력률에 의존.
- **(선택) 정렬 옵션** — 등록일/상태/금액/카테고리. 필터와 짝. **(선택) 택배함 검색**(메모/택배사/번호). 둘 다 부담 시 다음 마이너.
- **(선택·소) 대시보드 당겨서 새로고침** — 목록 화면과 일관.
- 위 "오늘 도착·모두읽음·휴지통 복구 딥링크" 3종은 비용 대비 가치가 높아 **v1.1 포함 권장**, 나머지는 후보. 최종 포함 여부는 Harness 설계 단계에서 확정. *(2026-06-23 정정: 최종 스코프 — '모두 읽음'은 포함, '휴지통 복구 딥링크'는 제외, '오늘 도착' 대시보드 카드는 구현 후 A1 로 **폐기**(4카드 축소).)*

### v1.1 추가 (2026-06-23 — 출시 후 시뮬레이터 스모크 발견 #1~#4)
> v1.1 구현·머지 후 시뮬레이터 점검에서 나온 UX 수정·추가 4건. 기존 제약(익명·로그인 없음·$0·개인정보 비영속) 불변. 구현은 harness `v1_1_0` step 20~24(구 phase 12~14).
- **#1 대시보드 빈 목록 = 항상 4카드**: 운송장 0건이어도 "등록 CTA 빈화면"이 아니라 4카드(카운트 0)를 그대로 표시(현황판이라 0이라도 보여줌, 휴지통·새 알림은 0 아닐 수 있음). 택배함(목록) 빈 상태 CTA는 유지 — 대시보드의 EmptyState만 제거.
- **#2 상세 "택배 정보"와 "수정" 분리**: 상세 헤더의 단일 연필을 **두 아이콘**으로 분리 — **수정**(연필 `Pencil`, 택배사·번호) / **택배 정보**(`FileText`, 메모·카테고리·금액). 인라인 박스 없음·헤더 진입 규칙은 유지.
- **#3 택배사 자동선택 보수화**: 추정 후보가 2개 이상이면 1순위를 자동으로 채우지 않고 드롭다운에서 명시 선택(오선택 방지, ADR-026). 후보 1개로 확실할 때만 자동.
- **#4 택배사·운송장번호 수정**: 상세 "수정"에서 식별자 편집 가능. 단 추적 식별자라 **클라이언트 재등록**으로 처리(`POST` 새 등록[먼저] → 택배 정보 이관 → `DELETE` 기존, 새 서버 엔드포인트 없음 — ADR-027). 변경 없음=no-op, 이미 추적 중이면 그 상세로. 식별자 변경 시 **새 번호로 새로 추적 시작**(이전 타임라인 교체)을 모달에 고지.

### 알림 정책 (변경 없음 + 기록 연동)
- 발송 규칙(단계 전환 1회·거래성만·**즉시 발송**[조용시간 폐지 ADR-030]·음소거 ADR-020). **음소거된 구독은 발송 제외 → 알림 기록에도 없음**(일관).
- 알림 기록은 발송된 알림의 **사후 열람**일 뿐 새 발송을 만들지 않는다(광고성 아님, ADR-018 무관).

### 데이터·프라이버시 영향 (스토어 신고 갱신)
- **로컬 전용**(서버 미전송): 메모·카테고리·금액, 휴지통 스냅샷, 알림 읽음 상태, 시작 화면·완료 숨기기 설정 → App Privacy/Data Safety의 **서버 수집 항목에 추가되지 않는다**.
- **서버 신규 저장**: `notifications`(택배사·끝4자리·상태 문구·발송시각 — **수령인/메모 없음 = 비-PII**). 개인정보처리방침의 수집 항목 설명에 "발송한 배송 알림 기록(상태 메시지)"를 반영(수령인·연락처·주소 비저장 원칙 불변). 보존기간(예: 90일) 명시.
- **라이브 방침 정정(선행 필수)**: 라이브 `app/src/content/privacyPolicy.ts` 의 "배송완료 감지 시 즉시 삭제" 문구가 ADR-005 개정(보관)과 모순 → v1.1 라이브 동기화 시 **"보관"으로 정정**(설계 SoT `PRIVACY_POLICY.md §6` 는 이미 정정됨). 출시된 공개 방침이 실제 동작과 달라 정확성·컴플라이언스 리스크라 v1.1 동기화의 **명시 작업**으로 둔다.
- 인앱 "모든 데이터 삭제"는 서버 `notifications` 행 + 모든 로컬 스토어(휴지통·정보·읽음·설정)까지 폐기(ADR-017 일관).

### v1.1.1 (2026-06-30 — UX·버그 수정 묶음 #1~#4 · 패치)
> v1.1.0 출시 후 사용자 보고 4건의 수정. 기존 제약(익명·로그인 없음·$0·개인정보 비영속 ADR-005) 불변. 구매내역 캡처는 v1.1.2로 분리(아래). 결정 SoT: ADR-032~035, UI_GUIDE "v1.1.1", ENGINEERING P-9. 버전 정책: ADR-035(패치 라인 유지·마이너 보류).
- **#1 수취인 이름 마스킹 게이트**: 상세 상단 "받는 분"은 이름이 **식별 가능할 때만** 표시한다. 택배사가 이름 대신 `"받는 분"` 같은 라벨이나 완전마스킹(`***`)을 보내면 줄 자체를 숨긴다. 부분 마스킹(`김**`·`이**`·`김*윤`)·실명은 사용자가 알아볼 수 있어 표시(ADR-032).
- **#2 페이지 제목 고정**: 설정 화면 제목이 스크롤돼 사라지던 것을 다른 화면처럼 스크롤 밖에 고정한다. 알림·상세 등은 이미 고정이라 변경 없음(ADR-033).
- **#3 모달 바깥 탭으로 닫히지 않음**: 입력 중 모달 바깥을 잘못 터치해 모달이 닫혀 입력이 날아가던 문제 수정 — 바깥 탭은 키보드만 접고, 모달 닫기는 취소 버튼/뒤로만(ADR-034).
- **#4 모달 키보드 가림 수정**: 키보드가 떠도 모달이 가용 영역 안에서 재중앙 정렬돼 가리지 않는다(윗 공간이 남으면 위로 올라옴). `KeyboardAvoidingView`(ADR-034·ENGINEERING P-9).
