## 1. 배경·목적

온라인 쇼핑몰의 **주문 상세/구매내역 화면을 캡쳐해 업로드**하면, 이미지에서 **상품명·가격·카테고리**를 자동 추출·분류하는 기능.

- 입력: 사용자가 직접 캡쳐한 스크린샷(카메라 사진 아님 → 디지털 폰트 → OCR 정확도 높음).
- 출력: `{ 상품명, 가격, 카테고리 }` (편집 가능한 초안으로 제시).
- ✅ **스코프 확정**: 신규 독립 기능이 아니라, **기존 v1.1 "택배 정보(메모·카테고리·금액)" 입력의 자동 채움 보조**다. 직접 입력과 캡처 업로드를 **병행**하고, 캡처가 채운 값은 **수정 후 저장** 가능. 진입점 = 송장 상세 "택배 정보" 모달(`app/app/shipment/[id].tsx`).

## 2. 확정된 결정사항 (이 대화에서 합의)

| # | 항목 | 결정 |
|---|---|---|
| D1 | 파이프라인 | **하이브리드**: ① 온디바이스 OCR → ② 클라이언트 PII 마스킹 → ③ Workers AI 분류 |
| D2 | 분류 모델 | **`@cf/openai/gpt-oss-120b`** (Cloudflare 호스팅 오픈웨이트 — OpenAI API 아님, Workers AI neuron 체계) |
| D3 | OCR 엔진 | iOS=Apple Vision / Android=ML Kit (온디바이스·무료·한국어). 실측은 macOS Vision(iOS 프록시)으로 검증 |
| D4 | 전송 데이터 | **이미지 아님 — 마스킹된 텍스트만** 외부 전송 |
| D5 | 개인정보 보호 | 3중: (a) OCR이 기기 내 처리 (b) 클라이언트에서 PII 마스킹 (c) Workers AI는 입력을 학습에 미사용. **D1에 PII 저장 금지(기존 CRITICAL 유지)** |
| D6 | 비용 기준 | Workers AI 무료 10,000 neuron/day. gpt-oss-120b 실측 ≈ 39 neuron/건 → **약 250건/일 무료(앱 전체 공유)** |
| D7 | 통합·진입·저장 | 신규 화면 아님 — 기존 송장 상세 "택배 정보" 모달에 "캡처로 채우기" 추가(직접 입력 병행). 기존 `ShipmentInfo`(memo·category·amount) **재사용**, 로컬 AsyncStorage(`unboxing.shipment_info`), **D1 미저장**. 카테고리 = 기존 `CATEGORIES` 고정 9개 |

### 왜 이 결정인가 (근거 요약)
- **온디바이스 OCR 사용 이유**: 진짜 무료(영구·무제한), PII가 기기를 안 떠남, 빠름(~1초). OCR은 "글자 읽기"만 정확(캡쳐라 거의 100%).
- **분류를 LLM이 맡는 이유**: 사이트마다 레이아웃이 달라 "어디가 상품명/가격인가"는 위치 규칙으로 불가능 → **의미 이해(LLM)** 가 흡수. OCR과 LLM은 경쟁이 아니라 분업(읽기 vs 분류).
- **이미지 직접 전송(비전 LLM) 대신 텍스트 경로인 이유**: 비전은 PII 노출·neuron 소모·레이턴시(4~10초) 모두 불리. 텍스트는 토큰 적고 PII 마스킹 가능.
- **gpt-oss-120b 채택 이유**: 실측에서 8B는 가격 오독·카테고리 오류, 120b는 전부 정답 + 가장 빠름(§4).
- **Claude/Gemini 대신 Workers AI인 이유**: Gemini 무료는 입력을 학습에 사용(한국은 예외지역 아님). Claude는 Max 구독으로 앱 호출 불가(API 종량제=신규 비용). Workers AI는 학습 미사용 + 무료 한도 + 기존 스택과 정합.

## 3. 아키텍처 (파이프라인)

```
[캡쳐 이미지]  (사용자 기기)
     │
     ▼
① 온디바이스 OCR  ── Apple Vision(iOS) / ML Kit(Android)
     │  텍스트 + 읽기순서       ※ 이미지가 기기를 떠나지 않음
     ▼
② PII 마스킹  ── 이름·전화·주소 제거 (클라이언트)
     │  마스킹된 텍스트만
     ▼  (HTTPS)
③ Workers AI  ── env.AI.run("@cf/openai/gpt-oss-120b", …)
     │  { productName, price, category } JSON
     ▼
[편집 가능한 초안 표시]  (사용자 확인/수정)
```

| 단계 | 위치 | 책임 | 실패 시 |
|---|---|---|---|
| ① OCR | 기기 | 이미지→텍스트(읽기순서 보존) | 텍스트 0줄 → "인식 실패, 다시 촬영" |
| ② 마스킹 | 기기(클라이언트) | PII 제거 후에만 전송 | 마스킹 규칙은 §6 |
| ③ 분류 | Cloudflare Worker | 텍스트→구조화 JSON | 타임아웃/한도초과/JSON 깨짐 → 수동 입력 폴백(§7, 미해결 #2·#7) |

- Worker는 **기존 `unboxing-worker`에 신규 엔드포인트 추가**(예: `POST /classify-purchase`). 상시 서버 아님(요청 시 실행) → "0원 서버" 원칙 유지.
- `wrangler.toml`에 `[ai]` 바인딩(`binding = "AI"`) 추가 필요.

### 출력 → 기존 필드 매핑 (`app/src/lib/info.ts`)
LLM 출력 `{상품명, 가격, 카테고리}` → 기존 `ShipmentInfo`:
- **memo ← 상품명** (모달 메모 100자 제한 적용)
- **amount ← 가격** (`parseAmount` 검증: 0 ≤ n < 10^10 정수)
- **category ← 카테고리** (반드시 기존 `CATEGORIES` 9개 중 하나로 강제)

자동 채움 후 기존 `setInfo(id, info, {store: infoStore})`로 로컬 저장 — **신규 저장 경로 없음**. (송장 ID 변경 시 기존 `transferInfo` 그대로 적용)

## 4. 실측 검증 결과 (이미 완료 — 증거)

세션 scratchpad에서 실제 파이프라인을 실호출로 검증함.

**(a) 합성 주문내역** — OCR 100% 정확, 8B도 정답:
`{ "productName": "삼성전자 갤럭시 버즈3 프로", "price": 189000, "category": "가전/디지털" }`
(판매가 199,000·할인 10,000 섞여 있어도 "총 결제금액" 정확 식별, PII 3줄 마스킹됨)

**(b) 실제 네이버페이 캡쳐** (정답: 백팩 / 60,190원 / 의류·패션) — 모델 비교:

| 모델 | 상품명 | 가격 | 카테고리 | LLM 속도 | 판정 |
|---|---|---|---|---|---|
| llama-3.1-8b-fp8 | ✅ | ❌ 60,**990**(오독) | ❌ 가전/디지털 | 3.4s | 부정확 |
| **gpt-oss-120b** | ✅ | ✅ **60,190** | ✅ **의류/패션** | **2.1s** | **완벽** |
| glm-4.7-flash | — | — | — | 4.1s | reasoning이 max_tokens 초과로 잘림 |

- **레이턴시 실측(end-to-end)**: OCR 1.1초 + LLM 2.1초 ≈ **3.2초**.
- **neuron 실측**: gpt-oss-120b 건당 input 782 / output 213 토큰 → ≈ **39 neuron/건** → 무료 한도로 **약 250건/일**.
- gpt-oss-120b는 reasoning 과정에서 광고 상품(아메리카노·면도기)을 배제하고 5종 금액 중 "총 결제금액"을 정확히 골랐음.

## 6. 개인정보 처리

- **원칙**: 이미지·PII는 기기를 떠나지 않는다. 외부엔 마스킹된 텍스트만. 추출 결과의 PII는 D1 저장 금지(기존 CRITICAL).
- **현재 마스킹(스파이크)**: 라벨(`받는사람|연락처|배송지|주소…`) + 전화번호 정규식 줄 제거.
- **🔴 실측에서 드러난 구멍**: 네이버페이 캡쳐에서 이름("제진명")·주소("경기도 안양시…")가 라벨 없는 줄이라 통과 → LLM 전달됨.
- **강화안 = 다층 방어(defense in depth)** — 어느 한 겹에 의존하지 않음:
  1. **섹션 구간 제거(주 방어)**: OCR 읽기순서에서 PII 섹션 헤더(`배송지`·`받는사람`·`수령인`·`배송정보`…) ~ 다음 안전 섹션 헤더(`결제정보`·`주문금액`…) 사이를 **통째 제거**. 라벨 없는 이름도 위치로 제거됨.
  2. **패턴 정규식(안전망)**: 전화·우편번호·주소(행정구역·도로명)·이메일·카드번호 줄 제거 — 구간이 놓친 경우 보강.
  3. **보수적 기본(과삭제)**: 분류엔 상품명·금액만 필요 → 애매하면 안 보냄.
  4. **출력측 안전망**: LLM은 `{상품명·가격·카테고리}`만 반환(PII 미출력) + Workers AI 비학습·D1 미저장 → 설령 잔존해도 즉시 폐기.
- **검증**: 마스킹은 순수 함수(텍스트→텍스트) → **test-first(TDD)**. 실제 캡쳐 OCR 픽스처로 "PII 줄 전부 제거 + 상품·금액 줄 보존"을 단언. 마스킹 후 **잔존 PII 자동 탐지 게이트**(전화·이메일·주소 정규식)로 누출 0 확인.
- **한계**: 쇼핑몰마다 섹션 헤더가 달라 100% 보장은 어려움 → 다층 + 과삭제 + 출력측 안전망으로 리스크 최소화, 실측으로 헤더 사전 보강(§8 #4).

## 7. UX 설계

- **진입·병행**: 송장 상세 "택배 정보" 모달(`shipment/[id].tsx`)에 **"캡처로 채우기"** 버튼 추가. 기존 직접 입력(메모·카테고리 칩·금액)은 **그대로 유지**(두 경로 병행).
- **자동 채움 → 수정 후 저장**: 캡처 → OCR→마스킹→LLM → 모달의 메모·금액·카테고리 칩을 자동 채움 → 사용자가 **기존 칩/필드 컴포넌트로 교정** → 기존 `saveInfo`로 저장(맞지 않는 값 수정 가능).
- **단계적 표시**: OCR 결과(상품명·가격)를 ~1초에 먼저 그리고, 카테고리만 "분류 중" 스피너로 비동기 채움 → 체감 레이턴시 6초대 → 1초대.
- **편집 가능한 초안**: LLM 결과는 확정이 아니라 **수정 가능한 폼**으로 제시(대기 시간을 확인 시간으로 전환).
- **폴백**: LLM 5초+ / 한도초과 / JSON 깨짐 → 카테고리 "미분류" + 수동 선택. 분류가 죽어도 등록 흐름은 안 멈춤.
